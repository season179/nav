How it works

Follow the current local path for one TUI message. This page names what exists now and what is still future work.

Current message path

sequenceDiagram actor User participant TUI as TUI (Go) participant Server as nav-server participant Harness as nav-harness participant Resolver as Model resolver User->>TUI: types message TUI->>Server: POST /rpc session.sendMessage Server->>Server: create run and message IDs Server->>Resolver: resolve default provider and model alt model settings resolve Server->>Harness: map minimal OpenAI-compatible chunks Harness-->>Server: model and message events Server-->>TUI: GET /sessions/{id}/events else settings missing or invalid Server-->>TUI: run.failed event end TUI->>TUI: render transcript or visible error

Step by step

  1. User types a message in the TUI. The Go frontend sends a session.sendMessage JSON-RPC call to POST /rpc on the local HTTP server.
  2. Server creates IDs. nav-server validates the session and text, then creates a run ID and message ID.
  3. Model settings resolve. The backend reads the configured default provider and model. If settings are missing or invalid, the run fails visibly.
  4. Events are mapped. The current prototype maps a minimal OpenAI-compatible stream through the harness event mapper. This exercises the typed event path before the full agent loop exists.
  5. TUI fetches session events. The Go client calls GET /sessions/{id}/events, using Last-Event-ID on later reads.
  6. Output renders. The TUI turns model deltas, completed messages, and errors into transcript rows and status text.

Current status

The local HTTP + JSON-RPC + SSE path is real enough for the TUI to send prompts and render typed backend events.

Tool approval, filesystem writes, persistent sessions, and the full multi-step agent loop are still future work. Keep docs and tests honest about that boundary.
