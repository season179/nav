# Flue — Context Management

Research target: `/Users/season/Personal/flue` (commit at time of reading: working tree on `main`).

Flue is "The Agent Harness Framework" — a TypeScript runtime that wraps an
upstream LLM agent loop (`@earendil-works/pi-agent-core`, backed by
`@earendil-works/pi-ai`) with sandboxing, session persistence, compaction,
roles, skills, and HTTP/SSE/webhook delivery. Read `README.md:5-15` and
`AGENTS.md:1-22` for the framework's self-description.

The runtime crate lives in `packages/runtime/src/`. All citations below are
against that tree unless otherwise marked.

---

## 1. Executive summary

- **Flue is a thin context-management layer over `pi-agent-core`.** Flue owns
  history (a parent-linked entry DAG), the system-prompt assembly,
  the built-in tools, compaction, sandbox plumbing, and per-call overrides.
  `pi-agent-core`'s `Agent` owns the actual provider call loop, tool
  dispatch, streaming, and stop-reason handling. Flue mutates that agent's
  `state.{systemPrompt,model,tools,thinkingLevel,messages}` per call, then
  calls `harness.prompt()` / `harness.continue()` / `harness.waitForIdle()`
  (`packages/runtime/src/session.ts:457-469`, `:944-979`,
  `:1658-1660`).
- **One session = one append-only DAG of `MessageEntry`, `CompactionEntry`,
  and `BranchSummaryEntry` nodes.** The "active path" — leaf-to-root — is
  reconstructed each call and rendered into the flat `AgentMessage[]` the
  provider receives (`session.ts:190-264`).
- **Compaction is opinionated and bounded.** Threshold trigger at
  `contextWindow - reserveTokens`; overflow trigger on provider-reported
  overflow; one structured summary replaces older history; a `keepRecentTokens`
  tail of verbatim messages is preserved. Summary is generated by an LLM call
  (same provider stack, configurable model) and persisted as a `CompactionEntry`
  in the durable tree, never destructively replacing history
  (`compaction.ts:307-462`, `session.ts:1351-1512`).
- **Prompt-cache breakpoints are NOT explicitly set by Flue.** No
  `cache_control` / cacheControl insertion was found in the Flue runtime
  (search results below). Whatever caching exists is whatever
  `pi-ai`/`pi-agent-core` do internally.
- **Cross-session persistence is pluggable via `SessionStore`.** Node default
  is in-memory; Cloudflare target stores entries in the agent's
  Durable Object SQLite state. Storage key: `agent-session:[instanceId,
  harness, sessionName]` (`harness.ts:200-202`,
  `cloudflare/session-store.ts:8-29`).
- **"Subagents" are `session.task()` calls** — a detached child `Session`
  with its own history but the same sandbox, capped at `MAX_TASK_DEPTH = 4`.
  Re-discovers `AGENTS.md` + skills from the child cwd; returns just text +
  optional structured `data` to the parent (`session.ts:80`, `:1016-1165`,
  `harness.ts:134-186`).

## 2. End-to-end turn trace

### 2.1 Request entry → handler

An HTTP request hits `handleAgentRequest` which selects sync / SSE / webhook
mode by headers (`runtime/handle-agent.ts:123-192`). It builds a `runId`,
parses the body, creates a `FlueContext` (per-target factory: Node uses
`process.env` + in-memory stores; Cloudflare uses DO env + DO SQLite store),
and ultimately calls the user's `default export` agent handler
(`runtime/handle-agent.ts:336-413`, `client.ts:49-194`).

The agent handler calls `ctx.init({ model, sandbox?, ... })`
(`client.ts:113-178`). `init()`:

1. Resolves the sandbox into a `SessionEnv` (default in-memory sandbox if
   omitted; `local()`, `daytona()`, CF Containers, etc. otherwise)
   (`client.ts:230-276`, `sandbox.ts:65-261`).
2. If the caller passed `cwd`, wraps the sandbox in `createCwdSessionEnv` so
   relative paths resolve against the sandbox cwd, not `/`
   (`client.ts:142-144`, `sandbox.ts:30-57`).
3. **Discovers per-session context** by calling `discoverSessionContext(env)`
   — reads `AGENTS.md` and `CLAUDE.md` from cwd, lists `.agents/skills/*`,
   captures a `readdir(cwd)` directory listing, and assembles the system
   prompt (`context.ts:188-209`, `client.ts:146`).
4. Returns a `Harness` configured with the resolved `AgentConfig`
   (`client.ts:162-173`, `harness.ts:28-50`).

`harness.session(name?)` returns (or creates) a `Session`
(`harness.ts:52-122`). Persisted state is loaded from
`SessionStore.load(storageKey)`; if missing, an empty `SessionData{ version:
3, entries: [], leafId: null, ... }` is saved (`harness.ts:88-102`,
`:208-218`).

### 2.2 `session.prompt(text, options?)` — one full turn

Calling `prompt()` returns a `CallHandle` (an abortable thenable)
(`session.ts:555-583`, `abort.ts`). Inside, `runOperation` ensures session
exclusivity (one operation in flight at a time —
`session.ts:1223-1237`), emits `operation_start`, then enters
`runPromptCall` (`session.ts:1591-1671`):

**(a) Resolve effective role.** Precedence: call > session > harness
(`session.ts:762-769`, `roles.ts:14-23`). Used for both system-prompt
overlay and per-call model/thinking selection.

**(b) Optional result tools.** If a `result:` valibot schema is set,
`createResultTools(schema)` produces a `finish`/`give_up` tool pair
exposed only for this call. Until either is called, the LLM loop will
re-prompt with `buildResultFollowUpPrompt()` for up to `MAX_FOLLOWUPS = 32`
turns (`result.ts:129-231`, `session.ts:1687-1725`).

**(c) `withScopedRuntime` overrides agent state for the duration of the
call** (`session.ts:940-980`). It saves the previous values of
`harness.state.{tools, model, systemPrompt, thinkingLevel}` then sets:

- `state.model` ← resolved per precedence call > role > agent
  (`session.ts:772-789`).
- `state.systemPrompt` ← `buildSystemPrompt(role)`:
  agent-config system prompt + (if role set)
  `<role name="...">\n${role.instructions}\n</role>`
  (`session.ts:838-845`). Note: role instructions are appended at call time
  and never persisted into history.
- `state.thinkingLevel` ← call > role > agent > `'medium'`
  (`session.ts:791-800`).
- `state.tools` ← builtin tools + connector custom tools + per-call
  `ToolDef`s + optional `finish`/`give_up` from the result bundle
  (`session.ts:956-971`). Custom tool names colliding with builtins or each
  other throw (`session.ts:873-895`).

**(d) Snapshot history cursors.** Before calling the agent loop:

- `beforeLength = harness.state.messages.length` — used to copy NEW messages
  back into the durable tree afterward (`session.ts:1632`).
- `beforeLeafId = history.getLeafId()` — used by `aggregateUsageSince` to
  sum every `usage` value the call (and any compaction it triggered)
  appended (`session.ts:1633`, `:1543-1554`).

**(e) Drive the agent.** Without a result schema:

```
await this.harness.prompt(args.promptText, args.images);
await this.harness.waitForIdle();
await this.syncHarnessMessagesSince(beforeLength, args.source);
await this.checkLatestAssistantForCompaction();
this.throwIfError(args.errorLabel);
```

(`session.ts:1658-1668`). `pi-agent-core` runs as many provider turns as
needed (tool calls included) until the assistant produces a final message,
emits `turn_start`/`message_update`/`tool_execution_*`/`turn_end` events
that Flue forwards through `eventCallback` (`session.ts:472-531`).

**(f) Persist new messages.** `syncHarnessMessagesSince` slices messages
appended since `beforeLength` from the volatile flat array and appends each
to the durable `SessionHistory` (`session.ts:1312-1317`,
`:276-291`). Then `save()` writes the entire `SessionData` to the
`SessionStore` (`session.ts:1319-1324`).

**(g) Post-turn compaction check.** `checkLatestAssistantForCompaction` →
`checkCompaction(lastAssistantMessage)`:

- If `isContextOverflow` on the last assistant message → drop that failed
  assistant, run compaction with `willRetry: true`, then `harness.continue()`
  + `waitForIdle()` for the retry (`session.ts:1343-1394`, `:1494-1505`).
- Else if `shouldCompact(contextTokens, contextWindow, settings)` → run
  threshold compaction (no retry — caller's data is already in hand)
  (`session.ts:1385-1393`, `compaction.ts:177-186`).

**(h) Return.**

```
return {
  text: this.getAssistantText(),
  usage: this.aggregateUsageSince(beforeLeafId),
  model: { id: resolvedModel.id },
};
```

(`session.ts:1664-1668`). `aggregateUsageSince` walks the active path since
`beforeLeafId`, summing assistant-turn `usage` + compaction-entry `usage` so
the caller sees the true cost including any background summarization
(`session.ts:1543-1554`).

### 2.3 Per-turn request shape (assembled by Flue, sent by `pi-agent-core`)

For each call, the provider receives:

```
{
  systemPrompt: <HEADLESS_PREAMBLE
                 + AGENTS.md (+ CLAUDE.md)
                 + "## Available Skills" registry
                 + Date + working dir + readdir listing
                 + optional <role name="X">role.instructions</role>>,
  model: <resolved model>,
  thinkingLevel: <resolved level>,
  tools: [read, write, edit, bash, grep, glob, task]
          + connector-supplied tools (replaces builtins if connector.tools is set)
          + per-call ToolDef[]
          + optional [finish, give_up],
  messages: <history.buildContext()
             = either the full active path,
               or [synthetic user "[Context Summary] ..." + post-cut path]>,
}
```

System-prompt source: `context.ts:146-185` (`composeSystemPrompt`),
`context.ts:143-144` (`HEADLESS_PREAMBLE`):

```ts
export const HEADLESS_PREAMBLE =
  'You are running in headless mode with no human operator. Work
   autonomously — never ask questions, never wait for user input. Make
   your best judgment and proceed independently.';
```

(`context.ts:143-144`)

Builtin tool list: `agent.ts:12-20` (`BUILTIN_TOOL_NAMES`).
Connector tool override path: `session.ts:898-918`.
Result-tool injection: `result.ts:129-231`.

## 3. Subsystem findings

### 3.1 System prompt

Composed once at session init (or task creation) by `discoverSessionContext`
in the **session's sandbox cwd**, not the host process cwd
(`harness.ts:104-119`, `:141`; `context.ts:188-209`,
`session.ts:444-469`).

Layered parts (`context.ts:146-185`):

1. `HEADLESS_PREAMBLE` (always first, single fixed sentence).
2. `AGENTS.md` (+ `CLAUDE.md` if present), concatenated with `\n\n`
   (`context.ts:44-56`).
3. `## Available Skills` list — `name + description` only. **Skill bodies
   are NOT inlined.** The model is told where to find each
   `.agents/skills/<name>/SKILL.md` and instructed to read it at call time
   (`context.ts:157-167`, `types.ts:17-28`).
4. `Date: ...` + `Working directory: <cwd>` + a non-recursive
   `Directory structure:` from `readdir(cwd)` (`context.ts:170-181`).

Per-call additions are not merged into the durable system prompt. Role
overlay (`<role name="...">instructions</role>`) is applied via
`withScopedRuntime` and reverted after the call (`session.ts:838-845`,
`:951`, `:977`). This is explicitly documented:

> "Role instructions are applied as call-scoped system prompt overlays,
> not injected into the persisted user message history." — `README.md:296`

### 3.2 Tool definitions

Default tools created by `createTools(env, ...)` in `agent.ts:45-56`:

- `read` (`agent.ts:58-118`): 2000-line / 50 KB head-truncation, dir
  listing fallback.
- `write` (`agent.ts:120-151`): creates parent directories.
- `edit` (`agent.ts:153-204`): exact-text replace, unique-match enforced.
- `bash` (`agent.ts:206-277`): timeout becomes both a `timeout` hint to the
  sandbox connector AND a composed `AbortSignal.timeout(...)` for
  signal-blind providers. On timeout returns a recoverable
  `exitCode: 124` result instead of throwing.
- `grep` (`agent.ts:342-395`): shells out to `grep -rn`, caps 100 matches
  × 500 chars/line.
- `glob` (`agent.ts:397-438`): `find -type f -name`, caps 1000 results.
- `task` (`agent.ts:279-321`): the framework-owned delegation tool.

Connector sandboxes can return their own tool list via
`SandboxFactory.tools(env, options)`; Flue ignores its defaults and only
appends `task` to the connector's list (`session.ts:907-918`,
`types.ts:692-703`). Connector tools cannot be named `task` (reserved) and
custom tools cannot collide with active builtins
(`session.ts:873-938`).

Custom `ToolDef` accepts JSON Schema-compatible parameters; the `execute`
function returns a string that becomes the tool-result text
(`types.ts:57-66`, `session.ts:855-871`). Throws in custom tools are
surfaced to the LLM as tool errors.

### 3.3 Message history (the durable tree)

`SessionHistory` (`session.ts:190-352`) holds three entry types:

- `MessageEntry { type: 'message', message: AgentMessage, source }` —
  recorded per user, assistant, and toolResult message
  (`types.ts:580-584`, `session.ts:276-291`).
- `CompactionEntry { type: 'compaction', summary, firstKeptEntryId,
  tokensBefore, details, usage }` — produced by compaction
  (`types.ts:586-599`, `session.ts:293-310`).
- `BranchSummaryEntry { type: 'branch_summary', fromId, summary,
  details }` — present in types and tree-building code but I found no
  caller producing them in the runtime tree. `appendBranchSummary` is
  defined but never invoked (`session.ts:312-324`, search for callers
  returned only the method itself).

Each entry has `parentId` linking to the previous leaf; `leafId` is the
DAG's current tip. The "active path" is leaf-to-root reversed
(`session.ts:214-222`). `buildContextEntries()` walks the active path; if
the most recent compaction node exists, it emits a synthetic user message
with `[Context Summary]\n\n${summary}` followed by entries from
`firstKeptEntryId` onward (`session.ts:245-261`, `:373-384`).

Storage: `SessionData { version: 3, entries[], leafId, metadata, createdAt,
updatedAt }` saved on every mutation (`session.ts:336-345`, `:1319-1324`).

### 3.4 Injected context

Per-call user message is composed by `buildPromptText` /
`buildSkillByNamePrompt` / `buildSkillByPathPrompt`
(`result.ts:39-95`):

- `prompt(text)`: the user's text, plus an optional **result footer** when
  a `result:` schema is set (`result.ts:13-20`):

  > "When the task is complete, call the `finish` tool with your final
  > answer as its arguments… If you determine that you cannot complete the
  > task… call the `give_up` tool with a clear `reason`."

- `skill('name')`: `Run the skill named "X".` + optional `Arguments: <json>`
  + optional result footer (`result.ts:39-55`).
- `skill('path/file.md')`: similar but with `The file can be found at
  <resolvedPath>.` (`result.ts:57-85`).
- `session.shell(cmd)`: synthesizes a user/assistant/toolResult triple as
  if the model had called bash itself, with synthetic provider bookkeeping
  fields (`session.ts:660-733`, `:1256-1310`). Env values are redacted in
  the recorded args; real values flow only to `env.exec`.
- `task(...)`: spawns a child session — see §3.10.

The session.shell synthesis is unusual: out-of-band shell calls appear in
the transcript as model-initiated tool calls so compaction sees them
uniformly (comment at `session.ts:663-669`).

### 3.5 Prompt-cache breakpoints

**Flue does not set cache breakpoints.** Searching the runtime for
`cacheControl`, `cache_control`, or `cache.?breakpoint` returns matches
only inside:

- `cloudflare/workers-ai-provider.ts` — reading `cached_tokens` from
  responses for the `cacheRead` counter (`:114`, `:151-154`).
- `cloudflare/gateway.ts` — Cloudflare AI Gateway's `skipCache` / `cacheTtl`
  request options (`:16-20`), which are HTTP-gateway-level, not
  provider-prompt-caching.
- Compat config for OpenAI completions format (`:42-59`):
  `cacheControlFormat: undefined`.

The framework's `usage.ts` and `PromptUsage` carry `cacheRead`/`cacheWrite`
fields, but they're populated from provider-reported usage (`usage.ts:55-69`,
`cloudflare/workers-ai-provider.ts:143-164`). Whatever prompt caching
happens is whatever the underlying `pi-ai` providers do automatically. There
is no Flue-level "system prompt is cacheable, history is cacheable up to
index N" annotation visible in the source.

This is consistent with the architecture: Flue only mutates
`harness.state.{systemPrompt, messages, tools, model}` and lets pi-ai
serialize and dispatch (`session.ts:944-979`).

### 3.6 Compaction trigger

Three triggers (`compaction.ts:1-8`, `session.ts:1351-1394`):

1. **Threshold** — `shouldCompact(contextTokens, contextWindow, settings)`:
   `contextTokens > contextWindow - reserveTokens`
   (`compaction.ts:177-186`). Called after every assistant turn finishes
   (`session.ts:1385-1393`). No retry — the caller already received their
   result.
2. **Overflow** — `isContextOverflow(assistantMessage, contextWindow)`
   returns true (re-exported from pi-ai; checked at
   `session.ts:1357`). Drop the failed assistant, compact, then
   `harness.continue()` + `waitForIdle()` to retry the turn
   (`session.ts:1359-1378`, `:1494-1505`). Guarded by
   `overflowRecoveryAttempted` so an overflow loop doesn't double-recover.
3. **Manual** — `session.compact()` (`session.ts:735-739`).

`contextTokens` source preference: provider-reported `usage.totalTokens`
falls through to `input + output + cacheRead + cacheWrite`
(`compaction.ts:75-77`).

Default settings (`compaction.ts:37-71`):

- `enabled: true`
- `reserveTokens: min(20000, model.maxTokens)`, then clamped to
  `floor(contextWindow / 3)` (min 1024) if reserve would consume ≥ half
  the window — safety floor for tiny-context models.
- `keepRecentTokens: 8000` (unchanged regardless of window size; the
  reasoning is in the doc comment at `:46-50`: "recent-context fidelity
  depends on the active work, not the window size").

`init({ compaction: false })` disables threshold compaction but **leaves
overflow recovery and `session.compact()` active**
(`session.ts:543-545`, `compaction.ts:182`, `types.ts:374-385`).

### 3.7 Compaction algorithm

`prepareCompaction` is pure (`compaction.ts:493-541`):

1. Boundary start: previous compaction's `firstKeptIndex` (1, since the
   summary occupies index 0), or 0 if first compaction (`:504`).
2. Find a cut point in `[boundaryStart, end)` whose accumulated tail
   tokens ≥ `keepRecentTokens` (`compaction.ts:426-462`,
   `findCutPoint`). Valid cuts: user or assistant messages — **never a
   `toolResult` message**, because cutting inside a tool exchange would
   leave a toolCall with no matching toolResult and break providers
   (`:401-411`).
3. If the cut lands on an assistant (mid-turn), walk backward to the
   nearest user message — that's the `turnStartIndex`. The "split turn"
   prefix between `turnStartIndex` and the cut is summarized **separately**
   from the older history (`:413-462`).
4. Extract `messagesToSummarize = messages[boundaryStart : historyEnd]`
   and (if split) `turnPrefixMessages = messages[turnStart :
   firstKeptIndex]` (`:514-517`).
5. Track file ops by walking assistant tool-call blocks for
   `read`/`write`/`edit` and merging in the previous compaction's
   `details.readFiles`/`modifiedFiles` (`:200-241`, `:519-529`).

`compact(preparation, model, apiKey, signal)`
(`compaction.ts:628-699`):

- If split-turn: two parallel `completeSimple` calls (`pi-ai`) — one for
  history with `SUMMARIZATION_PROMPT` or `UPDATE_SUMMARIZATION_PROMPT`, one
  for the turn prefix with `TURN_PREFIX_SUMMARIZATION_PROMPT`. Concatenated
  with `\n\n---\n\n**Turn Context (split turn):**\n\n...`
  (`:659-675`).
- Else: single summarization call (`:677-686`).
- Append a `<read-files>` / `<modified-files>` block to the summary
  (`:689-690`, `formatFileOperations`).

The summarization prompts (`compaction.ts:307-397`) are explicit, structured
markdown templates with `## Goal`, `## Constraints`, `## Progress`,
`## Key Decisions`, `## Next Steps`, `## Critical Context`. Update mode
preserves prior fields and rolls "In Progress" → "Done"
(`:345-382`). The system prompt is fixed
(`SUMMARIZATION_SYSTEM_PROMPT`, `:309-310`):

> "You are a context summarization assistant… Do NOT continue the
> conversation. Do NOT respond to any questions in the conversation. ONLY
> output the structured summary."

`maxTokens` for summarization: `min(0.8 * reserveTokens, 16000)` for the
main summary, `min(0.5 * reserveTokens, 16000)` for the split-turn prefix
(`:553`, `:595`). Reasoning effort defaults to `'high'` when the
summarization model supports it (`:569`, `:605`).

### 3.8 Compaction: where the summary lands

`session.history.appendCompaction({ summary, firstKeptEntryId,
tokensBefore, details, usage })` appends a `CompactionEntry` as a new node
linked to the current leaf (`session.ts:1468-1474`, `:293-310`). The
tree is **not** rewritten; old `MessageEntry` nodes survive but are no
longer on the active path the next time `buildContext()` runs (the active
path now ends at the compaction node, which prepends a synthetic user
message and resumes from `firstKeptEntryId`).

After append, `harness.state.messages = history.buildContext()` resets the
volatile flat array to the new compacted form (`session.ts:1475`).
The summary's `usage` is stored on the entry so `aggregateUsageSince` later
folds it into the triggering call's reported cost
(`session.ts:1543-1554`, `types.ts:592-599`).

### 3.9 Memory: in-session & cross-session

- **In-session state.** `SessionHistory` (DAG) + `metadata`
  (`Record<string, any>` used for things like `taskSessions[]` records,
  `harness.ts:150-156`, `session.ts:1326-1339`). The metadata bag is
  user-extensible but Flue itself only writes task-bookkeeping.
- **Filesystem.** Sandbox-backed `SessionEnv`/`FlueFs`. On
  `local()`/Daytona/etc. the FS persists with the underlying host or
  container. On Cloudflare with `cf-shell` sandbox, the workspace is a
  durable SQLite-indexed filesystem; `hydrateFromBucket(workspace, R2)`
  copies seed data on first boot (`README.md:75-103`,
  `cloudflare/shell-sandbox.ts`).
- **Cross-session persistence.** Pluggable `SessionStore`
  (`types.ts:608-612`):
  - Node default: `InMemorySessionStore` (`session.ts:159-173`) — lost on
    process exit unless the user provides their own.
  - Cloudflare default per README: "session data is backed by Durable
    Objects" (`README.md:275`). The CF `session-store.ts` shows the DO
    state path; the README notes the generated entry point uses DO SQLite
    by default (`cloudflare/session-store.ts:1-4`).
  - Storage key: `agent-session:[instanceId, harness, sessionName]` —
    JSON-stringified array (`harness.ts:200-202`).
  - Affinity key: `${instanceId}::${harness}::${sessionName}` — used as
    `pi-agent-core`'s `sessionId` for whatever upstream affinity
    semantics it implements (`harness.ts:204-206`,
    `session.ts:467-469`).
- **Cross-conversation continuity** is achieved by reusing the URL
  `<id>` — see `README.md:251-273`. New `<id>` = new agent instance =
  new sandbox + new session tree.

### 3.10 Subagents (`session.task(...)`)

Two surfaces (`README.md:281-294`, `agent.ts:279-321`):

- `session.task(prompt, options)` — programmatic, returns either
  `PromptResponse` (text + usage + model) or `PromptResultResponse<T>`
  (data + usage + model) when a `result:` schema is set
  (`session.ts:644-658`, `:1016-1165`).
- The `task` tool exposed to the LLM, which the model can invoke as a
  tool call (`agent.ts:279-321`).

Both ultimately call `runTaskExclusive` (`session.ts:1061-1165`). Behavior:

- **Depth cap.** `MAX_TASK_DEPTH = 4` (`session.ts:80`,
  `:1074-1076`).
- **Session isolation.** A new `Session` is created with name
  `task:<parent>:<taskId>` and a fresh empty `SessionData`
  (`harness.ts:134-185`). It does **not** see the parent's message
  history.
- **Sandbox sharing.** The child reuses `parentEnv`, optionally wrapped in
  `createCwdSessionEnv` if `cwd` was passed (`harness.ts:138-140`).
  Files and shell state are shared.
- **Per-task context re-discovery.** `discoverSessionContext(taskEnv)`
  re-reads `AGENTS.md` + skills from the child cwd, so the child's
  system prompt reflects its working directory
  (`harness.ts:141-146`).
- **Role inheritance.** Precedence: explicit task `role` >
  parent's effective role (`session ?? agent`)
  (`session.ts:1080`, `:1095`). Model and thinkingLevel pass through
  similarly: per-call > role > inherited (`session.ts:1116-1126`).
- **Reintegration.** When invoked as a tool, the child's full text reply
  becomes the tool result content; the parent only sees one
  `AgentToolResult` blob in its message stream
  (`session.ts:1004-1013`). The parent message history thus contains
  one tool exchange, not the child's transcript. Programmatically, `task()`
  returns the full text or structured `data`. The child session is closed
  (in-memory unset) after the call but its persisted state remains in the
  store (`session.ts:1158-1164`).
- **Bookkeeping.** Parent's metadata records `{ session, taskId,
  storageKey }` for the child, allowing recursive deletion via
  `deleteSessionTree` (`session.ts:1326-1339`, `:1744-1761`).

### 3.11 Token budgeting

- **Tokenizer.** Heuristic `chars / 4` over message content
  (`compaction.ts:105-148`, `estimateTokens`). Image content blocks
  count as a flat 4800 chars proxy (`:138-141`). Used only when
  provider-reported usage is unavailable.
- **Preferred path.** `estimateContextTokens` finds the latest assistant
  message with a `usage` field and uses `calculateContextTokens(usage)`
  (i.e. provider-reported `totalTokens` or sum of input/output/cacheRead/
  cacheWrite) plus heuristic estimates for messages **after** that point
  (`compaction.ts:150-175`). This is what feeds the cut-point search.
- **Per-turn split.** Reserve = up to 20 000 tokens (or model.maxTokens),
  clamped down for tiny windows. KeepRecent = 8000 tokens of verbatim
  tail. Pre-summary "context" = whatever the provider says was used.
- **Truncation/backpressure paths:**
  - Tool outputs: `read` truncates head to 2000 lines / 50 KB
    (`agent.ts:5-6`, `:100-110`); `bash` truncates tail to same
    (`:328-329`); `grep` 100 matches × 500 chars (`:6-8`).
  - Tool-result text fed to the summarizer is capped at 2000 chars per
    block (`compaction.ts:245-251`, `TOOL_RESULT_MAX_CHARS`).
  - Result-tool follow-ups: `MAX_FOLLOWUPS = 32` (`session.ts:1698`).
  - There is no preemptive truncation of the message array sent to the
    provider — the provider sees the entire active context. Backpressure
    is exclusively via compaction (threshold) and overflow recovery
    (retry).

## 4. Non-obvious design choices

1. **Skill bodies live on disk, not in the system prompt.** The skill
   registry tells the model only `name + description` and the path
   pattern; the model is expected to read the SKILL.md file with its
   `read` tool at use time. The doc-comment explains this lets users edit
   skills mid-session and lets relative references inside the skill
   resolve correctly from the skill's location
   (`context.ts:88-97`, `types.ts:17-28`).

2. **`session.shell()` is recorded as a synthetic LLM-driven bash call.**
   Out-of-band shell calls appear in the transcript as user-message
   "Run this shell command:" + assistant tool-call + tool-result, so the
   compactor treats them uniformly with model-issued calls
   (`session.ts:660-733`, `:1256-1310`). Synthetic
   provider-bookkeeping fields (`api: 'flue-shell'`, zeroed `usage`) are
   used so cost aggregation isn't double-charged.

3. **Append-only DAG with leaf pointers.** Compaction does not rewrite
   history — it appends a new `CompactionEntry` whose parent is the
   pre-compaction leaf, and from then on `buildContext` walks the new
   leaf. Old messages are still in `entries[]`, just off-path. This makes
   the data shape friendly to future branching but is currently used only
   for compaction (`session.ts:347-352`, `:245-261`).

4. **Compaction summaries are tightly templated.** The fixed-shape markdown
   (Goal / Constraints / Progress / Key Decisions / Next Steps / Critical
   Context) plus the `UPDATE_SUMMARIZATION_PROMPT` for subsequent
   compactions implies the second-and-later compactions explicitly preserve
   prior structure rather than re-summarize the summary
   (`compaction.ts:345-382`).

5. **Split-turn handling.** When the cut lands inside an assistant turn,
   Flue runs two summarization calls in parallel — one for the older
   history, one for the in-progress turn's prefix — and stitches them
   under a separator (`compaction.ts:659-675`). This avoids producing
   a context summary that ends mid-tool-call.

6. **Per-call state mutation.** `withScopedRuntime` mutates the shared
   `pi-agent-core` `Agent` state, runs, then restores. The session is
   single-flight (`runExclusive`), so this is safe in practice
   (`session.ts:940-980`, `:1223-1237`). It does, however, make the
   `Agent` instance look stateful in a way that surprises readers
   accustomed to per-call configs.

7. **Connector tools can fully replace the builtin tool set.** If a
   sandbox factory provides `tools(env, options)`, Flue uses ONLY that list
   plus `task` — no read/write/edit/bash/grep/glob (`session.ts:907-918`).
   Useful for sandboxes with different filesystem semantics (e.g.
   non-POSIX) but easy to forget.

8. **No `init()` re-entry per harness name.** Each request can call
   `init()` once per harness name; a duplicate throws
   (`client.ts:124-127`). The user can pass `name: 'X'` to get
   multiple isolated harnesses in one run.

9. **Headless preamble is short and load-bearing.** Single sentence at
   the top of every system prompt (`context.ts:143-144`), and a
   doc-comment explains its history: previously injected per-call into
   user messages, now hoisted to system to reduce redundancy
   (`context.ts:133-145`).

## 5. Under-developed or risky areas

1. **No explicit prompt caching strategy.** Flue rebuilds the system
   prompt on every `init()` (including a fresh `Date:` line and a fresh
   `readdir(cwd)` listing — `context.ts:170-181`), which **defeats
   prompt caching** if the cache key is over the system prompt. Two
   sessions started seconds apart will likely have different system
   prompts even when nothing else changed. This appears unintentional;
   the timestamp granularity and directory-listing instability are the
   culprits. (Caveat: the actual cache strategy is in `pi-ai`, which I
   couldn't read — `node_modules` is not installed at
   `/Users/season/Personal/flue/node_modules`. See open questions.)

2. **`branch_summary` is dead code.** `appendBranchSummary` is defined and
   `BranchSummaryEntry` is rendered by `pathToContextEntries`, but
   nothing in the runtime invokes it (`session.ts:312-324`,
   `:359-361`, `types.ts:601-606`). Either future-work that's
   slipped or a vestigial branch — worth confirming before relying on
   it.

3. **Heuristic tokenizer is rough.** The `chars / 4` estimator is
   conservative for English ASCII but loose for code, JSON, and non-Latin
   text. When the latest assistant lacks a `usage` field, all
   cut-point math is heuristic (`compaction.ts:105-148`,
   `:150-175`). For very small windows or atypical token-to-char ratios
   this can either over-compact or fail to compact in time.

4. **Reserve-vs-output asymmetry on tiny windows.** The safety floor
   (`reserveTokens ≤ contextWindow / 3`) leaves headroom but doesn't
   guarantee the compactor can actually emit its targeted summary —
   summarization `maxTokens` is `0.8 * reserveTokens` capped at 16 000
   (`compaction.ts:553`). On a 4k-window model the summary could be
   capped tighter than the kept tail.

5. **No mid-flight backpressure.** Once a turn starts, Flue does not
   intervene if `pi-agent-core` keeps emitting tool calls; the only
   stop conditions are `pi-agent-core`'s own iteration limit, the
   user's `signal`, or the provider returning `error`/`aborted`. The
   `MAX_FOLLOWUPS = 32` ceiling only applies to the result-schema retry
   loop (`session.ts:1698`).

6. **`SessionStore.save()` writes the entire `SessionData` blob each
   turn.** Long sessions accumulate every off-path entry too (compaction
   appends rather than rewrites). On the DO SQLite path this is fine for
   typical sizes but grows monotonically; there is no compaction-of-the-tree
   step that GCs orphaned `MessageEntry` nodes
   (`session.ts:1319-1324`, `:347-352`).

7. **Per-task `discoverSessionContext` re-reads `AGENTS.md` and walks
   `.agents/skills/`** for every task call. For deep task trees this
   re-reads the same files repeatedly (`harness.ts:141`). Likely
   fine on local FS, more expensive across networked sandboxes.

8. **Connector-tool override is all-or-nothing.** A connector cannot add
   one tool while keeping the framework defaults; providing `tools` means
   you own the entire list (`session.ts:907-918`). The comment in
   `session-store.ts:1-4` flags the connector boundary as the place to
   customize, but the ergonomics push you toward duplicating the default
   tool list in every connector.

## 6. Open questions / confidence gaps

- **Pi-agent-core internals.** I could not read
  `@earendil-works/pi-agent-core` or `@earendil-works/pi-ai` —
  `node_modules` is absent. Statements about what happens between
  `harness.prompt()` and `harness.waitForIdle()` (provider serialization,
  tool dispatch order, cache-control insertion) are inferred from Flue's
  usage sites and the comments at `session.ts:472-531`, not direct
  evidence. Likely high impact for the prompt-cache claim in §3.5.
- **Exact `contextWindow` source.** Used at `session.ts:1356`,
  `compaction.ts:184`. Comes from `Model.contextWindow` on the resolved
  pi-ai model. Whether models without metadata return 0 (skipping
  threshold) or a default is in pi-ai.
- **`isContextOverflow` semantics.** Imported from pi-ai and re-exported
  through Flue (`compaction.ts:10`, `:700`). The trigger condition
  (which providers/stop reasons it recognises) lives in pi-ai.
- **CF `runFiber` durability vs in-flight compaction.** README §3.9
  notes CF sessions are DO-backed and survive across requests. I did not
  trace what happens if compaction is mid-flight when a DO hibernates;
  `compactionAbortController` exists but the resume path isn't obvious in
  `handle-agent.ts`.
- **Whether `branch_summary` is actually unused.** I searched for callers
  of `appendBranchSummary` and `BranchSummaryEntry` and found only
  definitions and the renderer. Could be invoked from outside the
  `packages/runtime/src/` tree (e.g. via the public API by a host
  application), but no internal caller exists.
- **`storeResponses` semantics.** `applyProviderPayloadOverrides` sends
  `store: true` for OpenAI-Responses-style APIs when configured
  (`session.ts:823-836`). The product implications (server-side
  retention) are flagged but Flue doesn't reason about cleanup of those
  stored items anywhere I could find.
