# Pi Guardrails Research

Source project: `/Users/season/Personal/pi`. Citations below are repo-relative paths under that directory unless an absolute path is shown. Scope note: I inspected the repository source and examples, not Season's live `~/.pi/agent` configuration, so claims about what is enabled by default are source-path claims.

## 1. Executive summary

Pi is a mono repo for an extensible coding agent: the README identifies `pi-coding-agent` as the interactive CLI, `pi-agent-core` as the agent runtime, and `pi-ai` as the multi-provider LLM API (`README.md:19`, `README.md:23`, `README.md:24`, `README.md:25`). The main CLI path is `packages/coding-agent/src/cli.ts` -> `packages/coding-agent/src/main.ts` -> `createAgentSessionFromServices` -> `AgentSession.prompt()` -> `Agent.prompt()` -> `runAgentLoop()` (`packages/coding-agent/src/cli.ts:8`, `packages/coding-agent/src/cli.ts:10`, `packages/coding-agent/src/main.ts:590`, `packages/coding-agent/src/core/agent-session.ts:962`, `packages/agent/src/agent.ts:390`, `packages/agent/src/agent-loop.ts:95`).

The strongest built-in guardrails are structural rather than policy-heavy: explicit tool allowlists, tool argument validation, extension interception hooks, session append-only persistence, image blocking, auth-file permissions, provider/model config validation, retry/compaction, and abort handling (`packages/coding-agent/src/cli/args.ts:232`, `packages/agent/src/agent-loop.ts:578`, `packages/coding-agent/src/core/agent-session.ts:397`, `packages/coding-agent/src/core/session-manager.ts:700`, `packages/coding-agent/src/core/sdk.ts:290`, `packages/coding-agent/src/core/auth-storage.ts:60`, `packages/coding-agent/src/core/model-registry.ts:468`, `packages/coding-agent/src/core/agent-session.ts:2429`, `packages/coding-agent/src/core/agent-session.ts:1388`).

The core runtime is permissive by default for local execution: default active tools are `read`, `bash`, `edit`, and `write`; absolute paths are resolved and accepted; bash commands run through the user's shell; and there is no default destructive-command confirmation, filesystem sandbox, git dirty-worktree guard, or network/browser policy in the primary tool implementations (`packages/coding-agent/src/core/sdk.ts:280`, `packages/coding-agent/src/core/tools/path-utils.ts:44`, `packages/coding-agent/src/core/tools/bash.ts:79`, `packages/coding-agent/src/core/tools/write.ts:201`, `packages/coding-agent/src/core/tools/edit.ts:314`).

Pi's design pushes many guardrails into project instructions and extensions. `AGENTS.md` gives strict git/dependency rules and requires explicit confirmation before overriding that document (`AGENTS.md:37`, `AGENTS.md:45`, `AGENTS.md:56`, `AGENTS.md:171`). Optional example extensions demonstrate permission gates, protected paths, dirty-repo checks, sandboxed bash, and subagents, but they are loaded only through configured/CLI extension paths, not as unconditional core behavior (`packages/coding-agent/src/core/resource-loader.ts:394`, `packages/coding-agent/examples/extensions/permission-gate.ts:1`, `packages/coding-agent/examples/extensions/protected-paths.ts:1`, `packages/coding-agent/examples/extensions/dirty-repo-guard.ts:1`, `packages/coding-agent/examples/extensions/sandbox/index.ts:1`, `packages/coding-agent/examples/extensions/subagent/README.md:1`).

## 2. End-to-end guarded turn trace

1. Startup and guidance loading. `cli.ts` sets the process title/env and calls `main()` (`packages/coding-agent/src/cli.ts:12`, `packages/coding-agent/src/cli.ts:13`, `packages/coding-agent/src/cli.ts:20`). `main()` sets `PI_OFFLINE` and `PI_SKIP_VERSION_CHECK` when `--offline` or `PI_OFFLINE` is present, then parses args and exits on parse errors (`packages/coding-agent/src/main.ts:424`, `packages/coding-agent/src/main.ts:426`, `packages/coding-agent/src/main.ts:444`, `packages/coding-agent/src/main.ts:445`, `packages/coding-agent/src/main.ts:450`). `SettingsManager` loads global and project settings, and project settings take precedence through deep merge (`packages/coding-agent/src/core/settings-manager.ts:283`, `packages/coding-agent/src/core/settings-manager.ts:117`, `packages/coding-agent/src/core/settings-manager.ts:138`, `packages/coding-agent/src/core/settings-manager.ts:140`).

2. Session/cwd safety preflight. Before runtime services are created, `main()` resolves the final cwd because `--session` and `--resume` may select a session from another project (`packages/coding-agent/src/main.ts:496`, `packages/coding-agent/src/main.ts:501`, `packages/coding-agent/src/main.ts:506`). If a session from another project is opened by id, Pi prompts to fork it into the current directory before proceeding (`packages/coding-agent/src/main.ts:248`, `packages/coding-agent/src/main.ts:250`, `packages/coding-agent/src/main.ts:251`). If a session cwd is missing, interactive mode offers Continue/Cancel and non-interactive mode exits (`packages/coding-agent/src/main.ts:507`, `packages/coding-agent/src/main.ts:509`, `packages/coding-agent/src/main.ts:515`).

3. Resources and system prompt. The resource loader discovers `AGENTS.md`/`CLAUDE.md` first in the agent config directory, then from cwd ancestors root-to-leaf (`packages/coding-agent/src/core/resource-loader.ts:58`, `packages/coding-agent/src/core/resource-loader.ts:85`, `packages/coding-agent/src/core/resource-loader.ts:96`, `packages/coding-agent/src/core/resource-loader.ts:110`). 4. Session and tool setup. By default the active built-in tools are `read`, `bash`, `edit`, and `write`; `--tools` becomes an allowlist, and `--no-tools` can start with no tools (`packages/coding-agent/src/core/sdk.ts:280`, `packages/coding-agent/src/core/sdk.ts:281`, `packages/coding-agent/src/core/sdk.ts:282`). `--tools` is documented as applying to built-in, extension, and custom tools (`packages/coding-agent/src/cli/args.ts:232`, `packages/coding-agent/src/cli/args.ts:234`, `packages/coding-agent/src/cli/args.ts:235`).

5. User prompt preflight. `AgentSession.prompt()` first gives extension commands a chance to handle slash commands, then lets extension `input` handlers transform/handle the text, expands skill commands and prompt templates, rejects ambiguous streaming submission without `streamingBehavior`, validates model and auth, may compact before sending, and then creates a `role: "user"` message (`packages/coding-agent/src/core/agent-session.ts:967`, `packages/coding-agent/src/core/agent-session.ts:979`, `packages/coding-agent/src/core/agent-session.ts:998`, `packages/coding-agent/src/core/agent-session.ts:1005`, `packages/coding-agent/src/core/agent-session.ts:1024`, `packages/coding-agent/src/core/agent-session.ts:1041`, `packages/coding-agent/src/core/agent-session.ts:1057`). 6. Provider call. `Agent.prompt()` rejects overlapping active runs, normalizes input, and invokes `runAgentLoop()` with a context snapshot and loop config (`packages/agent/src/agent.ts:327`, `packages/agent/src/agent.ts:328`, `packages/agent/src/agent.ts:333`, `packages/agent/src/agent.ts:390`). `runAgentLoop()` appends prompt messages to context, emits lifecycle events, then streams an assistant response (`packages/agent/src/agent-loop.ts:103`, `packages/agent/src/agent-loop.ts:109`, `packages/agent/src/agent-loop.ts:116`, `packages/agent/src/agent-loop.ts:193`). 7. Tool call handling. 8. Turn continuation and recovery. Tool results are appended as `role: "toolResult"` and fed into the next loop turn (`packages/agent/src/agent-loop.ts:212`, `packages/agent/src/agent-loop.ts:727`). After each run, `AgentSession` may retry retryable provider errors with exponential backoff or compact context before continuing (`packages/coding-agent/src/core/agent-session.ts:918`, `packages/coding-agent/src/core/agent-session.ts:936`, `packages/coding-agent/src/core/agent-session.ts:950`, `packages/coding-agent/src/core/agent-session.ts:2447`). Print mode can emit JSONL events and exits nonzero for final assistant `error`/`aborted` stop reasons (`packages/coding-agent/src/modes/print-mode.ts:103`, `packages/coding-agent/src/modes/print-mode.ts:132`, `packages/coding-agent/src/modes/print-mode.ts:134`).

Alternative loops: `InteractiveMode`, `PrintMode`, and RPC are selected from the same runtime in `main()` (`packages/coding-agent/src/main.ts:678`, `packages/coding-agent/src/main.ts:681`, `packages/coding-agent/src/main.ts:707`). A newer `AgentHarness` class exists in `pi-agent-core` and also calls `runAgentLoop()`, but the inspected CLI path uses `AgentSession` rather than `AgentHarness` (`packages/agent/src/harness/agent-harness.ts:164`, `packages/agent/src/harness/agent-harness.ts:560`, `packages/coding-agent/src/main.ts:590`).

## 3. Guardrail subsystem findings

### Instruction hierarchy

Pi constructs a provider context with one `systemPrompt`, a message array, and tools; I did not find a separate runtime "developer" role in the primary loop path (`packages/agent/src/agent-loop.ts:292`, `packages/agent/src/agent-loop.ts:293`, `packages/agent/src/agent-loop.ts:294`, `packages/agent/src/agent-loop.ts:295`). A custom system prompt replaces the default prompt body, then appended prompt text and project context can be concatenated afterward (`packages/coding-agent/src/core/system-prompt.ts:53`, `packages/coding-agent/src/core/system-prompt.ts:56`, `packages/coding-agent/src/core/system-prompt.ts:60`). Default prompt construction appends project context files inside `<project_context>`/`<project_instructions>` tags and then appends visible skills if the read tool is active (`packages/coding-agent/src/core/system-prompt.ts:155`, `packages/coding-agent/src/core/system-prompt.ts:157`, `packages/coding-agent/src/core/system-prompt.ts:160`, `packages/coding-agent/src/core/system-prompt.ts:165`).

Project guidance has a human-level override rule: if the user's instructions conflict with `AGENTS.md`, the agent must ask for explicit confirmation before overriding (`AGENTS.md:171`, `AGENTS.md:173`). Runtime conflict handling is narrower: extension shortcut conflicts become diagnostics, and resource conflicts are reported while load-order precedence handles them (`packages/coding-agent/src/core/extensions/runner.ts:417`, `packages/coding-agent/src/core/extensions/runner.ts:422`, `packages/coding-agent/src/core/resource-loader.ts:403`, `packages/coding-agent/src/core/resource-loader.ts:404`). I found no semantic instruction-conflict resolver in the main prompt construction path; project guidance is concatenated as text, and any enforcement depends on the model, the user, or an extension hook (`packages/coding-agent/src/core/system-prompt.ts:159`, `packages/coding-agent/src/core/agent-session.ts:1074`, `packages/coding-agent/src/core/extensions/runner.ts:924`).

### Tool guardrails

The built-in tool set is explicit: coding tools are read/bash/edit/write, read-only tools are read/grep/find/ls, and `createAllToolDefinitions()` contains all seven (`packages/coding-agent/src/core/tools/index.ts:138`, `packages/coding-agent/src/core/tools/index.ts:147`, `packages/coding-agent/src/core/tools/index.ts:156`). CLI allowlists and disable flags are first-class (`packages/coding-agent/src/cli/args.ts:104`, `packages/coding-agent/src/cli/args.ts:108`, `packages/coding-agent/src/cli/args.ts:232`, `packages/coding-agent/src/cli/args.ts:234`). The registry filters built-ins, extension tools, and SDK custom tools through `_allowedToolNames`, and extension/custom tools can override built-in names in the final registry (`packages/coding-agent/src/core/agent-session.ts:2256`, `packages/coding-agent/src/core/agent-session.ts:2259`, `packages/coding-agent/src/core/agent-session.ts:2267`, `packages/coding-agent/src/core/agent-session.ts:2313`, `packages/coding-agent/src/core/agent-session.ts:2314`).

Tool arguments are schema-validated before execution, missing tools return error results, and extension `tool_call` hooks can block execution (`packages/agent/src/agent-loop.ts:569`, `packages/agent/src/agent-loop.ts:578`, `packages/agent/src/agent-loop.ts:580`, `packages/agent/src/agent-loop.ts:598`). `AgentSession` wires extension tool hooks directly into `Agent.beforeToolCall` and `Agent.afterToolCall`; if a `beforeToolCall` extension throws, Pi blocks execution by throwing an error (`packages/coding-agent/src/core/agent-session.ts:397`, `packages/coding-agent/src/core/agent-session.ts:403`, `packages/coding-agent/src/core/agent-session.ts:410`, `packages/coding-agent/src/core/agent-session.ts:414`).

There is no built-in per-command confirmation or command denylist in the primary bash tool implementation. The core bash tool spawns the configured shell with the model-supplied command, streams stdout/stderr, supports timeout/abort, and reports nonzero exit as an error (`packages/coding-agent/src/core/tools/bash.ts:79`, `packages/coding-agent/src/core/tools/bash.ts:95`, `packages/coding-agent/src/core/tools/bash.ts:105`, `packages/coding-agent/src/core/tools/bash.ts:375`, `packages/coding-agent/src/core/tools/bash.ts:397`). Confirmation is available through optional extensions: the permission-gate example blocks `rm`, `sudo`, and `chmod/chown 777` unless an interactive user approves, and it fails closed when no UI exists (`packages/coding-agent/examples/extensions/permission-gate.ts:1`, `packages/coding-agent/examples/extensions/permission-gate.ts:11`, `packages/coding-agent/examples/extensions/permission-gate.ts:19`, `packages/coding-agent/examples/extensions/permission-gate.ts:20`, `packages/coding-agent/examples/extensions/permission-gate.ts:25`, `packages/coding-agent/examples/extensions/permission-gate.ts:27`).

There is no default OS sandbox or network allow/deny policy in the core bash path. `--offline` disables startup network operations like update checks and package telemetry, not arbitrary bash network egress (`packages/coding-agent/docs/settings.md:53`, `packages/coding-agent/docs/settings.md:55`, `packages/coding-agent/src/main.ts:426`, `packages/coding-agent/src/main.ts:428`). The sandbox example demonstrates a loadable extension that overrides bash via `registerTool()` and uses `@anthropic-ai/sandbox-runtime` for filesystem/network restrictions, but its own comments describe it as an example and setup requires loading/copying the extension (`packages/coding-agent/examples/extensions/sandbox/index.ts:1`, `packages/coding-agent/examples/extensions/sandbox/index.ts:8`, `packages/coding-agent/examples/extensions/sandbox/index.ts:32`, `packages/coding-agent/examples/extensions/sandbox/index.ts:201`, `packages/coding-agent/examples/extensions/sandbox/index.ts:214`).

### Filesystem and git safety

Core file tools resolve `~`, absolute paths, and relative paths against cwd, but the resolver does not enforce a cwd boundary (`packages/coding-agent/src/core/tools/path-utils.ts:44`, `packages/coding-agent/src/core/tools/path-utils.ts:48`). `read` resolves the path and checks only readability before reading (`packages/coding-agent/src/core/tools/read.ts:241`, `packages/coding-agent/src/core/tools/read.ts:243`, `packages/coding-agent/src/core/tools/read.ts:244`). `write` resolves the path, creates parent directories, and writes the file; `edit` resolves the path, verifies it is accessible, reads it, applies exact replacements, and writes it back (`packages/coding-agent/src/core/tools/write.ts:201`, `packages/coding-agent/src/core/tools/write.ts:213`, `packages/coding-agent/src/core/tools/write.ts:218`, `packages/coding-agent/src/core/tools/edit.ts:313`, `packages/coding-agent/src/core/tools/edit.ts:314`, `packages/coding-agent/src/core/tools/edit.ts:327`, `packages/coding-agent/src/core/tools/edit.ts:351`).

Pi does protect against concurrent writes to the same file by serializing file mutation operations per resolved/real path (`packages/coding-agent/src/core/tools/file-mutation-queue.ts:28`, `packages/coding-agent/src/core/tools/file-mutation-queue.ts:32`, `packages/coding-agent/src/core/tools/file-mutation-queue.ts:51`, `packages/coding-agent/src/core/tools/file-mutation-queue.ts:54`). The edit tool also enforces exact, unique, non-overlapping replacements in its interface and execution flow (`packages/coding-agent/src/core/tools/edit.ts:299`, `packages/coding-agent/src/core/tools/edit.ts:304`, `packages/coding-agent/src/core/tools/edit.ts:347`).

Git safety is mostly project guidance, not core enforcement. `AGENTS.md` tells agents to stage explicit files, verify status, avoid `git reset --hard`, `git checkout .`, `git clean -fd`, `git stash`, `git add -A`, `git add .`, `git commit --no-verify`, and never force push (`AGENTS.md:47`, `AGENTS.md:51`, `AGENTS.md:52`, `AGENTS.md:56`, `AGENTS.md:58`, `AGENTS.md:64`). The core bash tool can run any shell command in cwd, so these git rules are not enforced by the default bash implementation itself (`packages/coding-agent/src/core/tools/bash.ts:79`, `packages/coding-agent/src/core/tools/bash.ts:375`). An optional dirty-repo guard extension checks `git status --porcelain` before session switch/fork and blocks or prompts, but it is an example extension rather than a mandatory core guard (`packages/coding-agent/examples/extensions/dirty-repo-guard.ts:1`, `packages/coding-agent/examples/extensions/dirty-repo-guard.ts:15`, `packages/coding-agent/examples/extensions/dirty-repo-guard.ts:28`, `packages/coding-agent/examples/extensions/dirty-repo-guard.ts:36`).

### Prompt-injection resistance

Pi has basic structural separation for prompt components, but no default trust-classifier for untrusted file/web/tool content in the primary loop. Project guidance is read from files and inserted directly into the system prompt text between XML-like tags (`packages/coding-agent/src/core/resource-loader.ts:63`, `packages/coding-agent/src/core/system-prompt.ts:157`, `packages/coding-agent/src/core/system-prompt.ts:160`). Skills are listed with XML escaping for name/description/location, but the skill file itself is later read by the model through normal tools when needed (`packages/coding-agent/src/core/skills.ts:342`, `packages/coding-agent/src/core/skills.ts:350`, `packages/coding-agent/src/core/skills.ts:363`).

Tool outputs are appended as `toolResult` messages and become part of the next context unless transformed by extensions (`packages/agent/src/agent-loop.ts:212`, `packages/agent/src/agent-loop.ts:727`, `packages/agent/src/agent-loop.ts:732`). Extensions can transform the full message context before provider conversion and can mutate the raw provider payload before it is sent (`packages/coding-agent/src/core/sdk.ts:375`, `packages/coding-agent/src/core/extensions/runner.ts:858`, `packages/coding-agent/src/core/extensions/runner.ts:871`, `packages/coding-agent/src/core/extensions/runner.ts:890`, `packages/coding-agent/src/core/extensions/runner.ts:904`). This is powerful for implementing injection filters, but also means extension code is inside the trust boundary.

I found no default browser/web restriction subsystem in the main loop. Browser automation does not appear as a built-in tool in `createAllToolDefinitions()`, and startup offline mode only covers startup network operations (`packages/coding-agent/src/core/tools/index.ts:156`, `packages/coding-agent/src/core/tools/index.ts:158`, `packages/coding-agent/src/core/tools/index.ts:164`, `packages/coding-agent/docs/settings.md:55`). Web-page prompt injection would therefore be mediated by whatever extension or bash/browser tool reads that content, not by a central untrusted-content policy in the core loop.

### Data protection

Credentials are stored in `auth.json` under the agent dir with parent directory mode `0700`, file mode `0600`, and lockfile coordination for reads/writes (`packages/coding-agent/src/core/auth-storage.ts:56`, `packages/coding-agent/src/core/auth-storage.ts:60`, `packages/coding-agent/src/core/auth-storage.ts:63`, `packages/coding-agent/src/core/auth-storage.ts:67`, `packages/coding-agent/src/core/auth-storage.ts:70`, `packages/coding-agent/src/core/auth-storage.ts:107`, `packages/coding-agent/src/core/auth-storage.ts:111`). Runtime API keys passed by CLI are not persisted (`packages/coding-agent/src/core/auth-storage.ts:223`, `packages/coding-agent/src/core/auth-storage.ts:227`). `getAuthStatus()` reports sources/labels without returning credential values (`packages/coding-agent/src/core/auth-storage.ts:346`, `packages/coding-agent/src/core/auth-storage.ts:349`, `packages/coding-agent/src/core/auth-storage.ts:354`).

Config values can intentionally execute shell commands when they start with `!`; stdout is used as the secret/config value and command results are cached for the process lifetime (`packages/coding-agent/src/core/resolve-config-value.ts:12`, `packages/coding-agent/src/core/resolve-config-value.ts:18`, `packages/coding-agent/src/core/resolve-config-value.ts:55`, `packages/coding-agent/src/core/resolve-config-value.ts:57`, `packages/coding-agent/src/core/resolve-config-value.ts:78`, `packages/coding-agent/src/core/resolve-config-value.ts:83`). This is useful for secret managers but broadens the config trust boundary.

Session data is append-only JSONL and includes messages after the first assistant message exists; entries are appended with raw `JSON.stringify()` (`packages/coding-agent/src/core/session-manager.ts:700`, `packages/coding-agent/src/core/session-manager.ts:843`, `packages/coding-agent/src/core/session-manager.ts:846`, `packages/coding-agent/src/core/session-manager.ts:853`, `packages/coding-agent/src/core/session-manager.ts:855`). Bash output is kept in bounded memory for display and full output is written to a temp file when truncated (`packages/coding-agent/src/core/tools/output-accumulator.ts:28`, `packages/coding-agent/src/core/tools/output-accumulator.ts:91`, `packages/coding-agent/src/core/tools/output-accumulator.ts:110`, `packages/coding-agent/src/core/tools/output-accumulator.ts:215`, `packages/coding-agent/src/core/tools/bash.ts:355`). I found no default redaction of secrets/PII before session persistence or temp-file output persistence in these code paths; the evidence is that persistence writes raw JSON entries and raw chunks to temp files (`packages/coding-agent/src/core/session-manager.ts:855`, `packages/coding-agent/src/core/tools/output-accumulator.ts:217`).

Telemetry is limited but not absent. Settings docs say `enableInstallTelemetry` controls anonymous install/update pings, while `--offline`/`PI_OFFLINE` disables startup network operations including update checks and install/update telemetry (`packages/coding-agent/docs/settings.md:53`, `packages/coding-agent/docs/settings.md:55`). Provider requests can include attribution headers for OpenCode, OpenRouter, and Cloudflare-related models unless install telemetry is disabled for the OpenRouter/Cloudflare header path (`packages/coding-agent/src/core/sdk.ts:134`, `packages/coding-agent/src/core/sdk.ts:138`, `packages/coding-agent/src/core/sdk.ts:141`, `packages/coding-agent/src/core/sdk.ts:145`, `packages/coding-agent/src/core/sdk.ts:153`).

### Runtime validation

Argument parsing reports diagnostics and exits on errors before runtime creation (`packages/coding-agent/src/main.ts:444`, `packages/coding-agent/src/main.ts:445`, `packages/coding-agent/src/main.ts:450`). Extension flags are validated against registered flags and unknown flags become runtime errors (`packages/coding-agent/src/core/agent-session-services.ts:85`, `packages/coding-agent/src/core/agent-session-services.ts:94`, `packages/coding-agent/src/core/agent-session-services.ts:115`, `packages/coding-agent/src/core/agent-session-services.ts:117`). `models.json` is validated with a schema and custom provider/model checks before custom models are accepted (`packages/coding-agent/src/core/model-registry.ts:465`, `packages/coding-agent/src/core/model-registry.ts:468`, `packages/coding-agent/src/core/model-registry.ts:514`, `packages/coding-agent/src/core/model-registry.ts:531`, `packages/coding-agent/src/core/model-registry.ts:553`).

At turn time, Pi validates model selection and auth before sending non-streaming prompts (`packages/coding-agent/src/core/agent-session.ts:1024`, `packages/coding-agent/src/core/agent-session.ts:1029`). Tool validation is centralized around lookup, argument preparation, TypeBox-style validation, before-hook blocking, execution, after-hook mutation, and error-result creation (`packages/agent/src/agent-loop.ts:569`, `packages/agent/src/agent-loop.ts:579`, `packages/agent/src/agent-loop.ts:580`, `packages/agent/src/agent-loop.ts:598`, `packages/agent/src/agent-loop.ts:656`, `packages/agent/src/agent-loop.ts:676`, `packages/agent/src/agent-loop.ts:710`).

Provider/runtime fallback paths include image blocking, retryable error detection, exponential backoff, abortable retry sleep, and compaction rather than retry for context overflow (`packages/coding-agent/src/core/sdk.ts:290`, `packages/coding-agent/src/core/sdk.ts:297`, `packages/coding-agent/src/core/agent-session.ts:2425`, `packages/coding-agent/src/core/agent-session.ts:2432`, `packages/coding-agent/src/core/agent-session.ts:2447`, `packages/coding-agent/src/core/agent-session.ts:2477`). ### Human-in-the-loop controls

Core human approval points are session/cwd safety and extension-driven UI, not default per-tool approvals. Cross-project session resume prompts ask whether to fork into the current directory, and missing-session-cwd recovery offers Continue/Cancel in interactive mode (`packages/coding-agent/src/main.ts:176`, `packages/coding-agent/src/main.ts:248`, `packages/coding-agent/src/main.ts:250`, `packages/coding-agent/src/main.ts:386`, `packages/coding-agent/src/main.ts:407`, `packages/coding-agent/src/main.ts:409`). Interrupt/resume behavior is robust in the TUI. Escape is the default `app.interrupt` binding (`packages/coding-agent/docs/keybindings.md:83`, `packages/coding-agent/docs/keybindings.md:85`). In interactive mode, Escape aborts streaming, aborts running bash, exits bash mode, or triggers tree/fork on double escape with an empty editor (`packages/coding-agent/src/modes/interactive/interactive-mode.ts:2379`, `packages/coding-agent/src/modes/interactive/interactive-mode.ts:2380`, `packages/coding-agent/src/modes/interactive/interactive-mode.ts:2382`, `packages/coding-agent/src/modes/interactive/interactive-mode.ts:2388`, `packages/coding-agent/src/modes/interactive/interactive-mode.ts:2393`). ### Subagents

Subagents are present as an example extension, not as a built-in core subsystem in the main tool set. The example says it spawns a separate `pi` process per subagent invocation with an isolated context window and JSON mode (`packages/coding-agent/examples/extensions/subagent/index.ts:1`, `packages/coding-agent/examples/extensions/subagent/index.ts:4`, `packages/coding-agent/examples/extensions/subagent/index.ts:12`). Its `runSingleAgent()` builds args with `--mode json`, `-p`, `--no-session`, optional model/tool allowlist, optional appended system prompt, then spawns a child process in the selected cwd (`packages/coding-agent/examples/extensions/subagent/index.ts:288`, `packages/coding-agent/examples/extensions/subagent/index.ts:289`, `packages/coding-agent/examples/extensions/subagent/index.ts:290`, `packages/coding-agent/examples/extensions/subagent/index.ts:317`, `packages/coding-agent/examples/extensions/subagent/index.ts:321`, `packages/coding-agent/examples/extensions/subagent/index.ts:329`).

Subagent guardrails are inherited only to the extent the child `pi` invocation is configured with tools/model/system prompt and whatever default runtime/extensions that child loads. Agent definitions can be user-level or project-level, with project agents found under nearest `.pi/agents`; when scope is `both`, project agents override same-named user agents (`packages/coding-agent/examples/extensions/subagent/agents.ts:97`, `packages/coding-agent/examples/extensions/subagent/agents.ts:101`, `packages/coding-agent/examples/extensions/subagent/agents.ts:102`, `packages/coding-agent/examples/extensions/subagent/agents.ts:106`, `packages/coding-agent/examples/extensions/subagent/agents.ts:108`). The README warns that project-local agents are repo-controlled prompts that can instruct the model to read files and run bash, defaults to user-level agents, recommends project scope only for trusted repos, and prompts before project-local agents in interactive mode (`packages/coding-agent/examples/extensions/subagent/README.md:55`, `packages/coding-agent/examples/extensions/subagent/README.md:57`, `packages/coding-agent/examples/extensions/subagent/README.md:59`, `packages/coding-agent/examples/extensions/subagent/README.md:61`, `packages/coding-agent/examples/extensions/subagent/README.md:63`, `packages/coding-agent/examples/extensions/subagent/README.md:65`).

## 4. Non-obvious design choices

1. Extension-first guardrails. Pi's core loop exposes `tool_call`, `tool_result`, `context`, `before_provider_request`, `before_agent_start`, and UI primitives, so guardrails can be layered without changing core tools (`packages/coding-agent/src/core/extensions/types.ts:1084`, `packages/coding-agent/src/core/extensions/types.ts:1127`, `packages/coding-agent/src/core/extensions/runner.ts:806`, `packages/coding-agent/src/core/extensions/runner.ts:858`, `packages/coding-agent/src/core/extensions/runner.ts:890`, `packages/coding-agent/src/core/extensions/runner.ts:924`). This makes policy pluggable but means a default install can be much more permissive than the example extensions suggest.

2. Extension tools can replace built-ins. `registerTool()` stores tools by name, and the registry adds built-ins first then extension/custom tools afterward, overwriting same-name entries (`packages/coding-agent/src/core/extensions/loader.ts:192`, `packages/coding-agent/src/core/extensions/loader.ts:194`, `packages/coding-agent/src/core/agent-session.ts:2313`, `packages/coding-agent/src/core/agent-session.ts:2314`, `packages/coding-agent/src/core/agent-session.ts:2315`). The sandbox example relies on that by registering a replacement `bash` tool (`packages/coding-agent/examples/extensions/sandbox/index.ts:8`, `packages/coding-agent/examples/extensions/sandbox/index.ts:214`).

3. Context files are highest-context but not mechanically protected. Global `AGENTS.md`/`CLAUDE.md` loads before ancestor project files, and all are concatenated into the system prompt (`packages/coding-agent/src/core/resource-loader.ts:85`, `packages/coding-agent/src/core/resource-loader.ts:91`, `packages/coding-agent/src/core/resource-loader.ts:110`, `packages/coding-agent/src/core/system-prompt.ts:155`). This is simple and transparent, but conflict semantics remain textual/model-mediated.

4. Full bash output can persist outside the session file. The bash result shown to the model is tail-truncated, but full output is saved to a temp file when truncated and the path is included in tool details (`packages/coding-agent/src/core/tools/bash.ts:279`, `packages/coding-agent/src/core/tools/bash.ts:345`, `packages/coding-agent/src/core/tools/bash.ts:355`, `packages/coding-agent/src/core/tools/output-accumulator.ts:215`). That helps recover diagnostics, but it is a data-protection boundary worth documenting.

5. Auth status avoids value disclosure, but config resolution can execute commands. Runtime status labels tell the UI where auth came from without returning secrets (`packages/coding-agent/src/core/auth-storage.ts:346`, `packages/coding-agent/src/core/auth-storage.ts:349`). At the same time, `models.json`/auth config values may run shell commands via `!`, which is a deliberate secret-manager affordance and a configuration trust risk (`packages/coding-agent/src/core/resolve-config-value.ts:12`, `packages/coding-agent/src/core/resolve-config-value.ts:18`, `packages/coding-agent/src/core/resolve-config-value.ts:55`).

## 5. Under-developed or risky areas

1. Default local execution is broad. The default active tools include bash and write/edit, paths can be absolute, and bash runs arbitrary shell commands in cwd with no default denylist or confirmation (`packages/coding-agent/src/core/sdk.ts:280`, `packages/coding-agent/src/core/tools/path-utils.ts:48`, `packages/coding-agent/src/core/tools/bash.ts:79`, `packages/coding-agent/src/core/tools/write.ts:201`, `packages/coding-agent/src/core/tools/edit.ts:314`).

2. Filesystem sandboxing is optional. The core file tools do not enforce cwd write boundaries, and the sandbox/protected-paths implementations are examples loaded only when configured (`packages/coding-agent/src/core/tools/write.ts:201`, `packages/coding-agent/src/core/tools/edit.ts:314`, `packages/coding-agent/examples/extensions/sandbox/index.ts:32`, `packages/coding-agent/examples/extensions/protected-paths.ts:1`).

3. Prompt-injection resistance depends heavily on prompting and extensions. Project files and tool outputs are inserted into the model context without a built-in untrusted-content classifier, and extension hooks can mutate both context and provider payloads (`packages/coding-agent/src/core/system-prompt.ts:160`, `packages/agent/src/agent-loop.ts:727`, `packages/coding-agent/src/core/extensions/runner.ts:858`, `packages/coding-agent/src/core/extensions/runner.ts:890`).

4. Session/temp persistence can capture sensitive data. Sessions append raw JSON message entries, and truncated bash output is written raw to a temp file (`packages/coding-agent/src/core/session-manager.ts:855`, `packages/coding-agent/src/core/tools/output-accumulator.ts:217`). I found no default redaction layer in those persistence paths.

5. Git and destructive-command policies are not runtime-enforced by default. The project guidance is strict, and example extensions show how to enforce some cases, but the core bash tool can run git/destructive shell commands unless a configured extension blocks them (`AGENTS.md:56`, `AGENTS.md:58`, `packages/coding-agent/src/core/tools/bash.ts:375`, `packages/coding-agent/examples/extensions/permission-gate.ts:13`, `packages/coding-agent/examples/extensions/dirty-repo-guard.ts:16`).

6. Subagent authority is easy to misunderstand. The example subagent tool uses a separate `pi` process and `--no-session`, but child agents still get configured tools and may run bash/read files depending on their agent definition (`packages/coding-agent/examples/extensions/subagent/index.ts:288`, `packages/coding-agent/examples/extensions/subagent/index.ts:290`, `packages/coding-agent/examples/extensions/subagent/README.md:57`). Project-local agents have an interactive confirmation path, but that disappears if `confirmProjectAgents` is false or there is no UI (`packages/coding-agent/examples/extensions/subagent/index.ts:499`, `packages/coding-agent/examples/extensions/subagent/index.ts:512`, `packages/coding-agent/examples/extensions/subagent/README.md:65`).
