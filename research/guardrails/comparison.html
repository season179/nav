Guardrails across harnesses

Side-by-side comparison of five coding agent implementations across permissioning, sandboxing, filesystem safety, prompt injection, secrets, human approvals, and subagent authority.

Sources: Claude Code / Codex / KimiFlare / OpenCode / Pi

Orientation
01Safety Posture
Where guardrails live
Layered
Every tool call passes through schema validation, tool validation, hooks, central permission resolution, and then execution. The design is strongest around Bash and permission precedence.
Sandbox plus approval
Sandboxing and approval are separate axes joined by presets. The tool orchestrator approves, runs sandboxed, and re-asks before any no-sandbox retry.
Mode-driven
Plan/edit/auto modes, hooks, loop caps, web-fetch caps, and output reducers do real work. Containment is weaker than the UX guardrails.
Permission rules
Every tool goes through declarative allow/deny/ask rules. Strong at prompting and path awareness, but not isolated from the host process.
Extension-first
Core has allowlists, validation, hooks, and append-only sessions. Harder safety policies are mostly optional extensions rather than mandatory defaults.
02Default Local Risk
Medium
Default prompts for uncovered writes and commands, but reads are broad and secret files inside the worktree are not specially protected.
Lower in sandboxed presets
Workspace/read-only profiles provide OS-backed write and network constraints. Full-access removes both human and sandbox review.
High in auto
Edit mode prompts, but auto mode allows mutating tools with no filesystem boundary or structural destructive-command block.
Medium-high
Default no-match behavior is ask, and outside-worktree paths ask, but approved shell commands run with user privileges and no sandbox.
High by default
Default active tools include read, bash, edit, and write. Paths may be absolute, and Bash has no built-in confirmation.
03Strongest Guardrail
Deny-first permissions plus a fail-closed Bash AST parser. Broad allow rules cannot override deny rules, and parser ambiguity forces prompting.
Runtime sandboxing with managed network constraints and optional Guardian auto-review. The no-sandbox retry requires fresh consent.
Operator control loop: plan mode blocks mutation, edit mode prompts, hooks can veto, and loop/web-fetch caps stop spirals.
Unified permission plane: native tools, shell, webfetch, read/write, MCP, and doom-loop checks all route through ctx.ask().
Extensibility: tool hooks can block or mutate calls, extension tools can replace built-ins, and example extensions show permission gates and sandboxes.
04Largest Gap
Untrusted content
No structural quarantine for file/web/MCP output, and no default secret-read block for files inside the working directory.
Git plus main-agent injection
No dirty-worktree, branch, push, or force-push layer. Guardian is hardened, but the main agent lacks untrusted-output framing.
Containment
No write boundary, no default sandbox for normal tools, no prompt-injection isolation, and auto mode relies on advisory text.
No sandbox
Permissioning is strong, but allowed commands run in the user's process space. No redaction or prompt-injection boundary.
Permissive core
The default loop can read, write, edit, and run shell commands broadly. Optional extensions can fix this, but default safety is thin.
Permissions
05Decision Center
Who says yes
Central engine
hasPermissionsToUseToolInner evaluates tool-wide and content-specific rules after hooks. Hook allow cannot bypass settings deny/ask.
Tool orchestrator
ToolOrchestrator::run computes approval need, executes under sandbox, and handles escalation with another approval.
Mode controller
decidePermission() maps plan/edit/auto to deny, prompt, or allow. Tool specs mark whether permission is required.
Ruleset evaluator
Permission.ask() finds the last matching rule. No match defaults to ask.
Allowlist plus hooks
CLI --tools and --no-tools filter the registry. Before/after tool hooks are the main policy insertion point.
06Rule Precedence
Deny > ask > allow
Flattened across sources, so managed-policy deny remains un-overridable by project/local allow.
Policy plus profile
Approval policy decides consent, permission profile decides sandbox. Managed network constraints cannot be loosened by user/project config.
Mode wins
Mode instructions are appended late and the permission controller enforces them. There is no broader rule-precedence lattice.
Last match wins
Rules are append-only and evaluated with findLast. User config merged later can override defaults.
Load order plus text
Resource conflicts report diagnostics, and project instructions are concatenated. Semantic conflicts depend on the model or extensions.
07Mode System
Rich
default, acceptEdits, plan, bypassPermissions, dontAsk, and ant-only auto/bubble modes.
Preset based
read-only, auto, and full-access combine approval policy with read-only/workspace/danger profiles.
Simple
plan denies mutation, edit prompts, auto allows all tools and relies on prompt guidance for destructive actions.
Agent based
Build, plan, explore, scout, general, compaction, title, and summary agents carry distinct permission rules.
Tool selection
Core mode is mostly tool availability, custom prompts, and extension behavior. Permission modes are not a central core concept.
08Human Approval UX
Mature
Per-tool dialogs support allow once, persist allow, deny, and feedback. Hooks can answer permission requests but cannot override deny/ask rules.
Structured
Exec, patch, MCP, network, and request-permissions approvals flow through one-shot channels. Decisions include approved for session and policy amendments.
TUI first
Edit mode prompts per mutating tool, with argument/diff previews and denial feedback. Ctrl+C/Esc resolves pending permission as deny.
Unified
Ask rules create deferred UI approvals. Users can approve once, approve always, reject, or reject with feedback.
Extension driven
Core approvals cover session/cwd prompts. Per-command confirmations require extension UI primitives; non-UI confirm returns false.
09Hooks and Plugins
Powerful but gated
PreToolUse, PostToolUse, UserPromptSubmit, Stop, PermissionRequest, and more. Project hooks require workspace trust.
Present, less audited
Input hooks can block prompts and hooks can influence approval, but the source report flags hook trust as a confidence gap.
Executor-level
Hooks live on ToolExecutor, so SDK, print mode, init turn, and standard loop share veto behavior.
Plugin hooks
tool.execute.before/after wrap tool execution, but the plugin system itself has no audited permission model.
Primary extension seam
Extensions can intercept input, tool calls, tool results, context, and provider payloads, and can replace built-in tools by name.
10Bypass / Full Access
Layered bypass
bypassPermissions still respects deny rules, content-specific asks, safety checks, and root/sudo restrictions.
True danger mode
full-access maps to approval Never plus disabled sandbox, so neither human nor Guardian review applies.
Auto means allow
Auto mode approves all tools. Destructive operations are only covered by a system-prompt instruction to pause and describe.
Session persistence
"Always" approvals persist within the session. External-directory access can be allowed broadly for the session.
Permissive baseline
There is no equivalent bypass switch because the default core already lacks per-command confirmation and sandboxing.
Filesystem, Shell, Network
11OS Sandbox
Delegated real sandbox
Seatbelt/bubblewrap configuration is in-tree; enforcement lives in external @anthropic-ai/sandbox-runtime.
Core primitive
macOS Seatbelt, Linux bubblewrap/seccomp, and Windows restricted-token modes are selected by the sandbox manager.
Code mode only
TypeScript code mode prefers isolated-vm, then falls back to node:vm. Normal Bash/file tools are not sandboxed.
No
No container, chroot, OS sandbox, or capability boundary for shell commands once permission is granted.
Optional example
Core tools have no sandbox. A sample extension can replace Bash with @anthropic-ai/sandbox-runtime.
12Filesystem Write Boundary
Strong for writes
Writes are bounded by original cwd/additional dirs, symlink expansion, path validation, and a sensitive-path safety check before acceptEdits.
Sandbox enforced
Patch and exec writes are assessed against writable roots, then executed under an OS sandbox when auto-approved.
No
write and edit resolve paths but do not enforce cwd/project boundaries.
Permission boundary
Read/write/edit/apply-patch call external_directory if outside worktree, then ask edit/read permission. Not a sandbox.
No
File tools resolve ~, absolute paths, and relative paths, then read/write/edit if the process has access.
13Secret File Reads
Weak
No default blocklist for .env, SSH keys, AWS credentials, or npmrc. Reads inside cwd are not prompted by safetyCheck.
Limited
Sandbox controls writes/network, not universal full-disk reads. Secret redaction exists for memory generation, not normal command output.
Weak
No file-layer secret-read protection. Memory redaction catches a few token patterns only after data reaches memory flow.
Pattern gate
*.env and *.env.* require approval by default. Other secret formats rely on permissions and user judgment.
Weak
Read checks readability only. No default secret or PII redaction before session/temp persistence.
14Shell Command Safety
AST fail-closed
Tree-sitter command parsing refuses to interpret ambiguous structure and falls back to prompt instead of auto-allow.
Narrow danger detector
Dangerous-command special detection covers rm -f/-rf. Many destructive git/system commands are not specially flagged.
Plan-only filter
Plan mode has a read-only Bash classifier. Auto/edit Bash runs through the normal permission path with no deeper parser.
Tree-sitter path analysis
Shell commands are parsed to extract file-affecting paths and trigger external-directory approval. Shell command itself is also asked by pattern.
No core filter
Bash spawns the configured shell with the model-supplied command. Permission-gate examples are optional extensions.
15Git Safety
Generic ask
Git push/commit/force-push are not hard-blocked. They rely on generic Bash prompting, warnings, and model instructions.
Absent
No dirty-worktree, protected-branch, commit/push, force-push, reset, or clean guard for user repos.
Mostly absent
No dirty-worktree or pre-push guard. The Bash tool injects a co-author trailer into commits, which is not a safety boundary.
Snapshot but no policy
Git-backed snapshots enable revert. Dirty-worktree, branch, push, and commit constraints are not enforced.
Guidance only
AGENTS.md can forbid destructive git commands, but core Bash can still run them unless an extension blocks them.
16Network and Browser
WebFetch controls
URL validation, forced HTTPS, domain preflight, same-origin redirects, size/time caps, host approvals, and sandbox egress config.
Managed network
Default-deny proxy supports allowlists and DNS-rebind defense. No client-side arbitrary browser/fetch tool; web search is hosted.
Usage caps
Web-fetch spiral guardrails cap per-turn, per-session, and repeat-domain fetches. No central OS/network egress sandbox.
Permissioned fetch
WebFetch validates HTTP/HTTPS, asks by URL pattern, limits response size to 5MB and timeout to 120s.
No central policy
--offline disables startup network operations, not arbitrary Bash network egress or extension/browser behavior.
17MCP / External Tools
Server approval
Project MCP servers need approval. Bypass auto-approval excludes project settings to reduce malicious-repo RCE.
Approved and remembered
MCP approvals can be session-remembered. MCP output is not untrusted-tagged; threat model remains a gap.
Mode blocked in plan
MCP tools are blocked in plan mode. In other modes, guardrails depend on adapter-supplied needsPermission.
Blanket permission
All MCP tools ask with patterns: ["*"], without per-argument granularity.
Extension trust boundary
External tools can override built-ins and mutate context/provider payloads. Powerful, but extension code becomes trusted code.
18Output Reduction
Safety-adjacent
Budgeted
Tool result budgets, history snipping, and microcompact reduce blowups; not a content-safety filter.
Size only
Rollout persistence can truncate exec output in extended mode, but no secret/content filter runs on normal output.
Artifact-backed
Per-tool reducers cap output and preserve full output as artifacts for expansion.
Truncation service
Output size is capped, but content is not sanitized or classified before reaching the model.
Bash tail plus temp file
Bash output is tail-truncated for display, with full output written to a temp file when truncated. Other paths are thinner.
Prompt and Data
19Prompt Injection
Mostly instructional
One prompt line tells the model to flag suspected injection. No structural wrapper for file/web/MCP/memory payloads.
Guardian only
Guardian reviewer is explicitly injection-hardened. The main agent lacks general untrusted-output tagging.
No explicit defense
Tool outputs, files, web pages, and memories are injected directly, aside from UTF-16 surrogate cleanup.
No explicit defense
File content, web content, MCP output, and instruction files are passed through without isolation markers.
No central classifier
Project guidance and tool outputs enter context directly. Extensions can add filters but are not default.
20Instruction Hierarchy
Layered with tension
System prompt is carefully assembled, but CLAUDE.md is framed both as overriding behavior and as maybe-relevant user context.
Explicit model-facing precedence
System/developer/user prompt beats AGENTS.md; deeper AGENTS.md beats shallower. Enforcement is prompt-level, not semantic code.
Simple layers
Static prefix, session prefix, KIMI.md, memories, and mode instructions. Conflict handling is model-mediated.
Concatenated
Environment, instruction files, skills, agent prompt, and model prompt are appended. No explicit conflict resolver.
Single system string
Global/project context files are concatenated into system prompt text; explicit override semantics live mostly in AGENTS.md text.
21Secret Redaction
Narrow
Telemetry avoids raw strings by type convention. Secret scanning is limited to team-memory/OAuth cases, not transcripts or tool output.
Memory only
redact_secrets is wired into memory generation, not rollout logs, exec output, or general telemetry.
Pattern based
Memory redaction covers AWS keys, GitHub tokens, OpenAI keys, and generic hex tokens. Custom tokens can slip through.
No runtime redaction
Tool outputs and logs are not scanned for secrets. .env approval is the main mitigation.
No default redaction
Session JSONL and temp output paths persist raw content in inspected code paths.
22Telemetry Privacy
Strong by default
Default analytics are metadata-only; prompt/tool content requires explicit OTEL env opt-ins.
Conservative
Telemetry redacts prompts by default and ships structured metadata when analytics is enabled/authenticated.
Mostly metadata
Logs exclude prompts/completions according to the report, but Cloudflare AI Gateway remains a network-side log plane.
Not fully verified
OpenTelemetry is experimental; stats/telemetry extent was listed as an open confidence gap.
Limited but present
Install/update pings and provider attribution headers exist unless disabled. Offline disables startup telemetry, not provider calls.
23Persistence at Rest
Plaintext
Project transcripts and prompt history are plaintext JSONL. Retention and disable switches exist; encryption does not.
Plain JSONL rollout
Sessions persist as rollout JSONL with item-type/size exclusions but no secret scanning. Credentials use keyring/file controls.
JSON plus SQLite
Sessions, logs, config, and memory DB persist locally. Logs prune after 7 days; no broad encryption claim.
SQLite and git stash
Sessions in SQLite, approvals per project, snapshots in git stash. No encryption at rest.
JSONL sessions
Auth files use 0700/0600 permissions, but sessions and temp outputs can contain raw tool/message data.
24Runtime Validation
Deep
Zod schemas, tool-specific validation, hook output schemas, model-refusal handling, fallback model, and max-output recovery.
Strong
Structured output strictness, patch safety assessment, retry handling, invalid-image sanitization, and Guardian strict JSON parsing.
Partial
JSON parse and structured ToolError codes exist, but per-tool parameter schemas are not enforced server-side beyond parsing.
Schema-first
Effect Schema validates every tool parameter; invalid input returns a model-visible tool result.
Solid basics
Arg parsing, extension flag validation, model registry schema checks, tool argument validation, retries, image blocking, and compaction fallback.
Autonomy, Recovery, Subagents
25Loop Guards
Recovery focused
Stop hooks, max-output-token recovery, fallback model, context collapse, and tool budgets. No highlighted doom-loop detector in this report.
Task controls
Retries, aborts, budget limits, and Guardian circuit breaker. The report does not describe a general repeated-tool loop detector.
Explicit
Third identical tool call blocks, all-blocked turns exhaust, default 50 iterations, and web fetches are capped per turn/session/domain.
Doom-loop check
Three identical completed tool calls trigger doom_loop permission. Agents may also define step limits.
Retry/abort
Retries, compaction, abort handling, and print-mode failure exits exist. No central repeated-tool guard was identified.
26Cancel and Interrupt
Threaded through
Esc/Ctrl+C abort running tools via an abort controller; aborts classify as user interrupts.
Turn abort model
Op::Interrupt aborts tasks and clears pending approvals/permissions after cancellation is observed.
Permission-aware
Ctrl+C/Esc interrupts the turn and resolves pending permission as deny.
Signal-aware
Session cancel propagates abort signals; shell processes are force-killed after 3 seconds.
Robust TUI aborts
Escape aborts streaming, running Bash, retry, and compaction states through session abort handlers.
27Rollback / Revert
User-facing rewind
/rewind can restore code and/or conversation to a prior point.
Diff tracking
Per-turn diff tracking and rollout recovery exist, but the report does not identify a first-class user revert command.
No undo
Edit precision prevents accidental partial edits, but there is no rollback mechanism after changes are applied.
Snapshot revert
Git snapshots let users revert or unrevert all agent changes since a message, when the session is idle.
Resume/fork, not undo
Session resume/fork and abort are present; no core file rollback was identified.
28Subagent Model
Built in
Subagents have separate sidechain histories, final-text reintegration, async modes, disallowed tool sets, and optional worktree isolation.
Inherited runtime
Spawned agents clone parent config, approval policy, shell env policy, and permission profile; approvals route back to the parent.
Remote workers
Workers run through HTTP with mission briefs. Guardrail inheritance for remote containers remains an open question.
Task sessions
Foreground/background task subagents run in separate sessions. Derived permissions prevent some plan-mode bypasses.
Example extension
Subagents are not core. The example spawns separate pi processes with selected tools/model/prompt.
29Subagent Authority Risk
Can self-elevate
Project agents can request broad tools/modes; runtime ceilings are disallow filters, per-call permissioning, and managed policy.
Parent mediated
Depth/thread caps are enforced, and child approvals bubble through the parent session.
Unclear remote ceiling
spawn_worker requires permission, but the remote agent's internal guardrails were not fully audited.
No sandbox
Permission inheritance helps, but subagents share filesystem/process space and background tasks have limited oversight.
Project-agent trust
Project-local agents can instruct Bash/file reads. Interactive confirmation exists, but no-UI paths are weaker.
30Most Useful Nav Borrow
Borrow deny-first rules, source-independent policy precedence, Bash AST fail-closed parsing, and typed telemetry invariants. Add stronger untrusted-content framing.
Borrow the sandbox/approval split, no-sandbox reapproval, managed network immutability, and Guardian-style reviewer. Add explicit git safety.
Borrow the small mode model, executor-level hooks, visible permission modal, loop/web caps, and artifact-backed output reducers. Do not copy auto without containment.
Borrow the unified ruleset model, external-directory checks, doom-loop permission, and snapshot revert. Add OS sandboxing and redaction.
Borrow the extension seams, built-in replacement mechanism, and simple source-visible tool flow. Do not make core safety depend on optional extensions.
