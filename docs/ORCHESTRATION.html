01Industry consensus

The shape that won.

By mid-2026 the marketing still says multi-agent, but the architecture that actually ships in production is narrower than the buzzword suggests. Anthropic, OpenAI, Cognition, Microsoft (Agent Framework / AutoGen), and LangChain converged on the same shape — one orchestrator owns the conversation, spawns short-lived isolated subagents, and reads back compressed summaries. Peer-to-peer chat and free-form group chat remain mostly research curiosities; production teams found context bleed and coordination loops killed them.

Why this matters Treat supervisor + worker + handoff as the load-bearing pattern. Treat everything else (mesh, swarm, free-form group chat) as a research mode you can prototype but shouldn't deploy without an exit plan.

The three structural alternatives — peer-to-peer, hierarchical trees, group-chat — still appear in papers, but production teams keep falling back to the same recipe:

CORE PRINCIPLE

one owner of context

A single orchestrator holds the full transcript. Subagents see only the prompt slice they need. This is the rule that prevents the context-window blowups that killed earlier mesh designs.

CORE PRINCIPLE

ephemeral workers

Subagents are short-lived and isolated. They get a bounded task, run, return a compressed result, then disappear. No long-running specialists holding state behind your back.

CORE PRINCIPLE

summary, not transcript

Workers return a structured summary — never their full chain-of-thought. The orchestrator integrates and decides next step. This is enforced structurally in well-built systems, not by convention.

Claude Code is the canonical example you can read in five minutes: the main agent plans and integrates, Task-spawned subagents do bounded work, each subagent has its own context window and tool surface, and the result that lands back in the parent's window is a short summary, not the subagent's transcript.

02Primitives

Three primitives, two protocols.

Beyond the pattern, three coordination primitives have stabilised across frameworks. They differ only in whether the caller blocks, and where the result lands.

handoff
Synchronous. The caller blocks until the worker returns. The default for "do this for me, then I'll keep going" — code review, test runs, focused analysis.
assign
Asynchronous, fire-and-forget. Result lands later — in a queue, on a board, or in another agent's inbox. Used for long-running work the orchestrator doesn't want to wait on.
send_message
Inbox-style peer delivery. Used when two equal agents need to coordinate, or when you want to leave a note for a future run. Rarer than the other two in coding contexts.

The protocols

Two standards joined the Linux Foundation AI & Agents Foundation in December 2025. They are orthogonal and worth keeping straight:

MCP — model context protocol

Standardises the tool surface an agent calls into. What functions exist, what arguments they take, what they return. nav already speaks this shape with its five tools.

A2A — agent-to-agent

Standardises inter-agent task delegation and collaborative workflows. Founding support from OpenAI, Anthropic, Google, Microsoft, AWS, and Block. Designed for the multi-vendor case.

Mental model MCP is the agent-to-tool protocol; A2A is the agent-to-agent protocol. A system can adopt either or both. Today most internal orchestrators reinvent A2A in private; the foundation push is about making cross-vendor delegation possible at all.
03Reality check

Why 40% of pilots fail.

Survey numbers from 2026 paint a sobering picture: more than half of organisations claim multi-step agent workflows in production, but a large minority of those pilots quietly die within six months. The failure modes rhyme.

In production
57%

of organisations report multi-step agent workflows running in production today.

Six-month failure
40%

of multi-agent pilots are shut down within six months of going live.

Primary cause
3·

root causes account for nearly all of them — shared mutable state, runaway context, no retry budget.

The three failure modes

FM-01

shared mutable state without an owner

Two agents writing to the same file or memory blob with no arbiter. Looks fine in a demo; produces a corrupt working tree the moment two tasks run in parallel.

FM-02

unbounded context growth

Each handoff bolts another transcript onto the parent. Costs climb, latency climbs, the model starts losing the plot mid-paragraph. Compression isn't optional at scale.

FM-03

no circuit breaker on retries

A bad task gets respawned forever. The bill arrives. Or worse, the spawn loop wedges the queue and nothing else runs. Every working orchestrator caps retries and routes give-ups to humans.

The pattern Every production-grade system you read about — Hermes, Vibe Kanban, Cline — has an answer to all three. Every failed pilot has an answer to none of them. The list is short and well-known; the discipline to enforce it is rare.
04Hermes

The board as the truth.

hermes-agent at ../hermes-agent is the most production-shaped implementation of these ideas in the sibling repos. It departs from the textbook supervisor/worker shape in one important way: there is no in-process subagent tree. Instead, the board itself is the message bus.

Read this for gateway/run.py — function _kanban_dispatcher_watcher, around line 3813. The whole orchestrator is a 60-second tick loop.

The architecture in one paragraph: a single SQLite file at ~/.hermes/kanban.db (or one per board for multi-tenant setups) holds every task, every state transition, and every historical attempt. A dispatcher loop inside the gateway wakes up on a fixed interval, scans the board for tasks in the ready column, atomically claims the ones it can promote, and spawns each as a fresh subprocess — a full hermes -p <profile> chat -q invocation with environment variables pinning it to one task on one board. The worker runs to completion, calls kanban_complete() or kanban_block() to report back, and exits. The dispatcher picks up the result on the next tick.

Why this matters Most multi-agent systems use a queue or message bus as their coordination layer, with state spread across in-memory workers. Hermes inverts that: the durable board is the coordination layer. Workers are stateless leaves that pick up a task, do work, write a result, and die. If the dispatcher restarts, nothing is lost. If a worker crashes, the claim expires and the task gets reclaimed.

The DAG part task_links(parent_id, child_id) encodes a dependency graph. The dispatcher's recompute_ready() promotes a child from todo to ready only when every parent is done. This is a pull barrier — children wait upstream, no explicit retry loops needed.
06The card

The task card.

Each row in the tasks table is a card. Most fields you can guess from the column name. Five of them are doing real work.

@dataclass
class Task:
    id: str                          # user-provided or UUID
    title: str
    status: str                      # one of VALID_STATUSES
    assignee: str | None             # hermes profile name to spawn
    priority: int

    # ── coordination ─────────────────────────────────────
    claim_lock: str | None           # claimer id, set by CAS
    claim_expires: int | None        # TTL, default 15 minutes

    # ── isolation ────────────────────────────────────────
    workspace_kind: str             # "scratch" | "worktree" | "dir"
    workspace_path: str | None

    # ── circuit breaker ─────────────────────────────────
    max_retries: int | None          # per-task override
    consecutive_failures: int        # auto-blocks at limit

    # ── audit ────────────────────────────────────────────
    current_run_id: int | None       # FK into task_runs
    skills: list | None             # JSON; force-loaded skills
    result: str | None
The five load-bearing fields claim_lock + claim_expires = lockless coordination via CAS. workspace_kind + workspace_path = parallel-safe filesystem isolation, the answer to FM-01. consecutive_failures + max_retries = circuit breaker, the answer to FM-03. current_run_id = pointer into task_runs for forensic replay. skills = tool surface narrowing per task — workers don't see what they shouldn't touch.

Around the task there's an audit-grade ring: task_links for the DAG, task_comments for human/worker annotations, task_runs for every attempt (each claim opens one), and task_events for granular activity (claim_rejected, heartbeat, completed, blocked). Crash forensics work the same way they would on a database — there's no separate logging system to keep in sync.

07Coordination

Coordination without a broker.

No queue, no message bus, no zookeeper, no Redis. SQLite WAL plus one UPDATE statement does all the work.

The atomic claim

Every dispatcher tick races every other dispatcher and every other host pointing at the same database. The only thing preventing two workers from claiming the same task is a compare-and-swap inside a single SQLite statement:

UPDATE tasks
   SET status        = 'running',
       claim_lock    = ?,
       claim_expires = ?,
       started_at    = COALESCE(started_at, ?)
 WHERE id = ?
   AND status = 'ready'
   AND claim_lock IS NULL;   -- ← the CAS check

If rowcount == 1, this dispatcher won the race and may spawn. If it's 0, somebody else got there first; move on. SQLite's BEGIN IMMEDIATE serialises writers on the WAL lock, which means the whole game reduces to "whoever writes first wins" — there's no logical race window to design around.

WIP enforcement

One number caps live concurrency. From dispatch_once() in gateway/run.py around line 3947:

if max_spawn is not None and running_count + spawned >= max_spawn:
    break

Set kanban.max_spawn: 4 in config and you get four parallel tasks, system-wide. The dispatcher counts already-running tasks plus what it spawned this tick, and stops when the sum hits the cap. Nothing more clever than that.

Failure handling

When a spawn fails, times out, or crashes, the dispatcher increments consecutive_failures. After failure_limit (default 5, overridable per task via max_retries) the task is auto-blocked with the last error attached. There is no exponential backoff. The 60-second tick acts as the backoff.

What's missing No heartbeats from dispatcher to workers. No leader election. No service discovery. The board is the only thing both sides talk to, and SQLite's WAL provides the only synchronisation primitive. Multi-host setups shard by board (one DB per board) rather than running coordinated replicas.

Observability comes free

Three tables — tasks (current state), task_runs (every attempt with outcome), task_events (every transition with payload) — make the whole system inspectable with plain SQL. There's no ELK stack, no tracing back-end, no separate audit log. You write one query and you can see every claim, every reject, every retry, every block, for every task that ever existed.

08Landscape

The coding-agent kanban scene.

Hermes isn't alone. A small genre of tools wraps CLI coding agents — Claude Code, Codex, Copilot, OpenCode — behind a board UI. Most converge on the same recipe: a column-per-state board, per-task git worktrees to keep parallel agents from colliding, a WIP cap on the In Progress column, and a bounded iteration counter on the human-review loop.

The iterative kanban pattern

A task gets handed to an agent, the agent produces output, a human reviews it, sends it back, the agent revises, the human approves — or maybe sends it back again. The board is the loop, not a checklist. — Iterative Kanban Pattern for AI Agents, MindStudio

The conventions that have settled across all of them:

  • WIP cap on In Progress: 1–3 tasks. Surfaces bottlenecks; prevents the cost runaway.
  • WIP cap on Awaiting Human Review: 5–10 per reviewer. If it overflows, you either need more humans or a better first-pass model.
  • Iteration ceiling: 3–5 revision cycles before forcibly escalating. Without this, agents spin forever on tasks that need a human conversation.
  • Per-task worktree: the universal answer to "how do two agents avoid corrupting each other's working tree".
09For nav

What nav could borrow.

None of this is in nav today, and it doesn't have to be. nav is an interactive coding agent — one turn, one user, one model. But several of the pieces sit a single migration away, because the substrate is already correct.

What's already in the right shape

  • SQLite session store at ~/Library/Application Support/nav/nav.db. A tasks table next to session, event, and turn would be three migrations from "session log" to "session log + kanban board".
  • AgentEvent as wire format. Workers calling back via a structured event is exactly the "compressed summary" return shape — nav already enforces it for the agent loop, and could reuse the same shape for sub-task results.
  • Workspace-only writes. The path sandbox in tools/fs.rs is the same primitive hermes calls workspace_kind=worktree. nav's version is single-workspace today; making it per-task is a parameter, not a redesign.
  • NDJSON event stream for non-Rust frontends. A dashboard or board UI that consumes the same events the TUI does is a "build a different reducer" problem, not a protocol problem.

The real design question

OPTION A

in-process subagents

Spawn workers as async tasks inside the same nav process. Lower spawn latency. Shared tool implementations. Easier debugging. Worse crash isolation — a panic in a worker tears down the parent.

OPTION B

subprocess workers (hermes shape)

Each task is a full nav invocation with environment pins. Trivially safe — crashes are bounded. Higher spawn latency. Cross-process state handoff is harder, but the board can be the handoff.

A reasonable rule of thumb For interactive "do this for me right now" work — code review, focused refactor, test runs — option A. For unattended, queued, or overnight work — option B. The two can coexist on the same agent: a session is option A; a queued background board is option B.

If we ever do build this

The minimal viable kanban for nav is much smaller than hermes:

  1. A task table in the existing SQLite db. Fields: id, title, body, status, workspace_path, claim_lock, claim_expires, consecutive_failures, current_run_id.
  2. The same CAS-claim pattern. SQLite is already on the bench.
  3. One state machine: todo → ready → running → done|blocked. Skip triage until there's a reason; skip archived until the table needs pruning.
  4. A single --queue flag that makes a nav invocation a one-shot worker against an existing board row, instead of an interactive session.
  5. WIP cap = 1 until we have a second one working at all. Pin kanban.max_spawn = 1 by default; the moment it works the real cap becomes "how many model contexts the user is willing to pay for in parallel".
End of note These are field notes, not a plan. The honest answer to "should nav do this?" is "only if there's a use case the current single-session loop can't already serve". The current loop is good. The board pattern is good. They serve different shapes of work.
