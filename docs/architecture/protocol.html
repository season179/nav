nav/architecture/protocol

Protocol.

A stable multi-frontend contract for nav clients. The TUI is one client of this boundary, not the owner of it.

Two channels

Commands go in via JSON-RPC. Ordered session events stream out via SSE. Every frontend speaks the same contract.

Contract boundary

The protocol is the stable language between nav's backend and every frontend: terminal TUI, web UI, API client, and agent-to-agent client. It is defined by wire shapes, event ordering, replay semantics, and negotiated capabilities.

Frontend-specific rendering, keyboard handling, scrollback, optimistic UI, and reconnect UX stay outside this contract. Backend-owned work such as sessions, runs, models, tools, approvals, and filesystem effects stays behind the protocol.

Protocol and storage

JSON-RPC and SSE are frontend-facing protocol projections. They are not the canonical transcript ledger and must not become a provider-native history format smuggled through the UI boundary.

The future durable source of truth is the session store described in plans/session-storage.md: sessions have runs, runs produce ordered turns, and turns are made of parts. The backend owns writes to that ledger; clients send commands and render projected events.

Long-term invariants

  • Store canonical turns and turn parts in SQLite, then encode them for the selected provider on read.
  • Decode provider responses into canonical turns and parts before appending them to the ledger.
  • Keep provider_response_id, raw provider SSE chunks, and provider_state as correlation or continuation cache, never as the source of truth.
  • Treat SSE replay as replay of retained protocol events. Transcript reconstruction, fork, compaction, and future snapshots come from canonical session rows.
  • Keep model selection in session settings, such as settings_json.modelRef, so later clients are not tied to process-global or TUI-only configuration.

JSON-RPC commands

Clients send commands to POST /rpc with Content-Type: application/json. Requests and responses use JSON-RPC 2.0. JSON-RPC errors are returned as JSON response bodies, normally with HTTP 200; transport errors such as unknown paths still use HTTP status codes.

{
  "jsonrpc": "2.0",
  "id": "019f2f6f-f178-7a72-9f28-000000000001",
  "method": "session.sendMessage",
  "params": {
    "sessionId": "019f2f6f-f178-7a72-9f28-000000000100",
    "text": "Explain this repo"
  }
}

Envelope

  • jsonrpc must be the string 2.0.
  • id is a lowercase UUIDv7 request ID. The client normally generates it; the server may generate one only when parsing fails before an ID is available.
  • method is one of the stable method names below.
  • params is omitted only when a method explicitly permits no params.

Response

  • Successful responses include result and omit error.
  • Failed responses include error and omit result.
  • The response id matches the request id whenever the request ID parsed successfully.
Method Params Result Semantics
initialize { clientName, clientVersion?, protocolVersion?, clientKind?, capabilities? } { serverName, serverVersion, protocolVersion, capabilities, methods, events } Negotiates protocol version and optional client/server capabilities before session work.
session.create { cwd?, source?, settingsJson? } or omitted { sessionId } Creates a backend-owned session, retains session metadata outside the protocol event log, and appends session.created.
session.sendMessage { sessionId, text } { sessionId, runId, messageId } Appends user input to a session, starts a run, and acknowledges the run IDs. Completion is reported later over SSE.
run.cancel { runId } { runId } Requests cancellation for a running run and appends run.cancelled when accepted.
tool.approve { approvalId } { approvalId, accepted: true } Allows a pending tool approval created by tool.approval_requested.
tool.reject { approvalId, reason? } { approvalId, accepted: false } Rejects a pending tool approval and records an optional human-readable reason.
session.close { sessionId } { sessionId } Closes the session for new runs. Existing retained events remain replayable while the session is retained.

SSE session events

Clients subscribe to GET /sessions/{sessionId}/events. Each SSE frame includes an id line, an event line, and a JSON data payload. The SSE id must equal the payload event_id; the SSE event must equal the payload type.

id: 019f2f6f-f178-7a72-9f28-000000000102
event: model.text_delta
data: {"event_id":"019f2f6f-f178-7a72-9f28-000000000102","session_id":"019f2f6f-f178-7a72-9f28-000000000100","type":"model.text_delta","run_id":"019f2f6f-f178-7a72-9f28-000000000101","message_id":"019f2f6f-f178-7a72-9f28-000000000103","delta":"hello","metadata":{"provider_id":"compatible-gateway","configured_model_id":"vendor/model-large"}}

Envelope

  • event_id is a lowercase UUIDv7 and orders events within a session.
  • session_id identifies the owning session.
  • type is the event name. Event-specific fields are flattened into the same object.
  • All ID fields use the same lowercase UUIDv7 format as nav-types.

Replay and live mode

  • Without Last-Event-ID, the stream starts with the earliest retained session event and then continues live.
  • With a known Last-Event-ID, the stream resumes strictly after that event and then continues live.
  • With a malformed event ID, the server returns HTTP 400.
  • With an unknown session, the server returns HTTP 404.
  • With a valid but no-longer-retained event ID, the server returns HTTP 409 and a JSON resync error instead of an SSE stream.

Resync response

A 409 response is not an SSE frame. It tells the client to discard its local protocol projection and rebuild before reconnecting.

{
  "error": "event_not_retained",
  "sessionId": "019f2f6f-f178-7a72-9f28-000000000100",
  "lastEventId": "019f2f6f-f178-7a72-9f28-000000000080",
  "earliestEventId": "019f2f6f-f178-7a72-9f28-000000000120",
  "snapshotRequired": true
}

Until a storage-backed snapshot or session-view method exists, servers should retain enough events to replay active sessions from the beginning. Once trimming is enabled, snapshotRequired: true means the client must load the canonical session view before opening a fresh event stream.

Versioning and errors

Capability negotiation

initialize is the compatibility gate for future clients. Clients declare a protocol version and optional capabilities; the server returns the version it will speak, supported methods, supported event types, and enabled capabilities. Unknown capability keys must be ignored rather than treated as fatal.

{
  "clientName": "nav-tui",
  "clientVersion": "26.5.12",
  "protocolVersion": 1,
  "clientKind": "tui",
  "capabilities": {
    "sseReplay": true,
    "normalizedMessages": true,
    "toolApprovals": true,
    "providerMetadata": true
  }
}

The stable server capability keys are sseReplay, normalizedMessages, toolApprovals, fileEvents, providerMetadata, and sessionClose. A client must not assume a capability unless the server returns it as enabled.

Current implementation status

Already implemented

  • session.create, session.sendMessage, and run.cancel are routed by crates/nav-server/src/http/mod.rs.
  • session.create accepts optional source and settingsJson session metadata so the protocol event log can sit beside the future canonical SessionStore.
  • GET /sessions/{sessionId}/events returns retained events encoded by crates/nav-server/src/http/sse.rs.
  • crates/nav-server/src/http/event_store.rs keeps the in-memory protocol event projection used for replay and live subscriber fan-out.
  • IDs are validated as lowercase UUIDv7 strings by crates/nav-types/src/lib.rs.
  • The Go TUI client creates a session, sends messages, tracks Last-Event-ID, and parses SSE in tui/internal/client/client.go.

Prototype limitations

  • initialize, tool.approve, tool.reject, and session.close are declared but not yet routed by the HTTP server.
  • The method table above is the stable target. Declared-but-unrouted RPC structs still need wire-shape alignment before exposure, including capability negotiation fields and camelCase approval/session params.
  • The SSE endpoint currently returns a finite replay response and closes; the underlying event store has live subscribers, but the HTTP transport has not yet been held open.
  • If Last-Event-ID is valid but unknown, the current server returns HTTP 409 so clients can resync from a future canonical session view.
  • Protocol events are retained in memory only. They are delivery projections, not the canonical transcript ledger; crates/nav-harness/src/sessions/mod.rs is still a stub, so durable transcript storage, fork, compaction, and canonical replay remain backend storage work.
  • The current run path is intentionally minimal and maps model output through a synthetic OpenAI-compatible stream before the full harness loop is wired.
  • Local auth token types exist, but the current HTTP bootstrap exposes only baseUrl.
  • The current TUI consumes model.text_delta directly. Future general-purpose frontends should prefer normalized message.delta once emitted.

Transport

Local clients talk to HTTP on 127.0.0.1 with a random port. Frontend launchers may spawn the backend and discover its base URL from the bootstrap line written by nav-backend serve-http.

All protocol IDs are canonical lowercase UUIDv7 strings — they're not secrets, but they do leak approximate creation time, so authentication uses separate random tokens.
