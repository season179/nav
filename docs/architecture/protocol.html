Protocol

Two channels

Commands enter through JSON-RPC. Events are encoded as SSE so frontends can replay from Last-Event-ID.

JSON-RPC methods

Frontend commands hit POST /rpc. The current HTTP server handles the session and run methods first.

  • session.create: start a session
  • session.sendMessage: user input
  • run.cancel: abort a running agent loop

The protocol crate already names initialize, tool.approve, tool.reject, and session.close. Those are not wired through the local HTTP handler yet.

SSE event types

Events stream to GET /sessions/{id}/events. Reconnect with Last-Event-ID.

  • session.created
  • run.started / run.completed / run.failed
  • model.text_delta / model.reasoning_delta
  • message.delta / message.completed
  • tool.call_requested / tool.call_started / tool.call_completed
  • tool.call_delta / tool.approval_requested
  • provider.error / error
  • file.changed

Transport

The current local backend binds to 127.0.0.1 on a random port and writes a backend.ready JSON line with its base URL. The Go client can spawn that backend or connect to an endpoint passed by tests.

All protocol IDs are canonical lowercase UUIDv7 strings. They are not secrets; they can reveal approximate creation time, so future auth must use separate random tokens.
