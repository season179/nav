nav/backend/protocol event log

Protocol event log.

Implemented in memory. crates/nav-server/src/http/event_store.rs stores frontend-facing EventEnvelope projections for replay and fans out appended events to live subscribers.

The log is scoped to protocol delivery. It powers GET /sessions/{sessionId}/events, Last-Event-ID replay, and long-lived SSE streams.

session.sendMessage now creates a run, starts a background model-run worker, resolves the configured OpenAI-compatible model, and publishes provider stream chunks through this log as typed protocol events.

Run state is explicit: new runs begin as running, then settle as completed, failed, or cancelled. Provider-shaped failures emit provider.error before the terminal run event; transport failures go straight to run.failed.

It is not the canonical transcript ledger. It does not own turns, parts, compaction, forks, provider continuation state, cost totals, or durable recovery.

Session metadata such as source and settingsJson is retained on the session record, not copied into every protocol event.

The future canonical store remains the SQLite SessionStore planned in plans/session-storage.md: sessions contain many runs, runs are the agent-loop unit, runs produce ordered turns, and turns are made of turn_parts.

The persistence seam attaches after live provider chunks have been decoded: the same mapped deltas that feed SSE can later append or patch canonical assistant turn parts. Provider continuation state may be cached beside the run, but it must not become authoritative history.

Current HTTP replays retained events first, then keeps the connection open for live fan-out without making the TUI the owner of event semantics.
