How it works.

Follow one message from keystroke to completed run. Every interaction with nav follows this pattern.

The flow

Step by step

  1. User types a message in the TUI. The Go frontend sends a session.sendMessage JSON-RPC call to POST /rpc on the local HTTP server.
  2. Server creates a Run. nav-server receives the RPC, creates a Run in nav-harness, and enters the agent loop. The command response acknowledges immediately — it does not wait for the run to finish.
  3. Agent loop starts. The harness loads context (files, conversation history, skills), sends the prompt to the configured LLM provider, and begins receiving streaming tokens.
  4. Tokens stream back via SSE. Each chunk becomes a message.delta event pushed through GET /sessions/{id}/events. The TUI renders text as it arrives.
  5. Tool call may be requested. If the LLM decides to use a tool (read a file, run a command, etc.), the harness emits tool.approval_requested over SSE. The TUI shows the request to the user.
  6. User approves or rejects. The TUI sends tool.approve or tool.reject via JSON-RPC. If approved, the harness executes the tool and feeds the result back into the agent loop.
  7. Run completes. When the LLM finishes (or the user cancels), the harness emits run.completed. All events carry UUIDv7 IDs for replay via Last-Event-ID.

Current status

The prototype currently uses stdio between TUI and backend with a simple hello check — not the full HTTP + JSON-RPC + SSE protocol described above.

The next backend milestone is replacing that stdio path with the architecture shown here. The docs describe the target state.
