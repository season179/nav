TUI startup comparison
Sibling read: ../codex

Codex starts a product; nav starts a loop.

Codex's TUI startup is an operational boot path: resolve config, choose or create a thread, recover state, and enter a rich event loop. nav already has the important bones, but its TUI should grow only where that makes daily use feel continuous and recoverable.

What Codex Does

Codex starts by normalizing CLI flags into runtime configuration, then resolving Codex home, config, environment, model provider, auth requirements, telemetry/logging, and the local state database before the main app loop is entered. It also has explicit recovery handling around local state database startup failures in the CLI wrapper.

Boot

`codex_tui::run_main` parses config overrides, finds Codex home, probes the app-server target, derives environment paths, loads config, resolves provider defaults, initializes telemetry/logging, and opens the state database before handing off to the ratatui app.

Session

Startup then chooses a fresh, resumed, or forked thread. Resume can target an explicit id, `--last`, or a picker, and Codex can prompt when the saved session cwd differs from the current cwd.

Once inside `App::run`, Codex bootstraps the app server, obtains model and account status, starts/resumes/forks the server-side thread, builds the `ChatWidget`, enqueues the initial thread state and turns, refreshes startup skills, and prefetches rate limits without blocking the first useful frame.

Evidence: `../codex/codex-rs/tui/src/lib.rs:781-940`, `../codex/codex-rs/tui/src/lib.rs:1357-1629`, `../codex/codex-rs/tui/src/app.rs:642-990`, `../codex/codex-rs/cli/src/main.rs:1986-2018`.

What nav Does

nav's startup is smaller and easier to follow. The CLI canonicalizes the launch cwd, discovers skills from that cwd, opens the SQLite session store, either creates a new session or loads events for `--resume`, loads auth, builds the OpenAI transport, and enters the TUI only when stdout is a terminal and no headless flag is set.

Boot

The TUI enters alternate screen/raw mode, installs a panic hook to restore the terminal, builds slash-command entries, creates the chat widget, shows a welcome cell for fresh sessions, and replays durable resume events into the scrollback.

Turn Loop

The event loop draws history, composer, and status bar; handles submit/clear/quit/skill-queue actions; spawns `run_agent` for each prompt; and rejects a second prompt while a turn is running.

nav does have persistence: durable `AgentEvent`s are written to the session store, and `--resume` rebuilds the Responses input from stored events. But in the current TUI that rebuilt transcript is consumed only for the first prompt after resume; later prompts start from their own text. The scrollback also ignores streaming deltas and paints assistant text only when the final message arrives.

Evidence: `crates/nav-cli/src/main.rs:23-64`, `crates/nav-tui/src/lib.rs:101-162`, `crates/nav-tui/src/lib.rs:204-275`, `crates/nav-tui/src/widget.rs:46-55`, `crates/nav-core/src/agent.rs:111-180`.

What nav Is Missing

  • Conversation continuity beyond startup resume. nav can replay a saved transcript into the first resumed turn, but the TUI comments say later prompts do not maintain server-side conversation state.
  • Session UX. nav has `--resume <id>` and `--list-sessions`, but not a TUI picker, `--last`, fork flow, cwd prompt, or visible resume guidance comparable to Codex.
  • Operational startup handling. auth, DB, config, and startup failures mostly bubble as errors. Codex has more explicit local-state recovery and auth/onboarding paths; nav should add clear, local fixes where failures are common.
  • Daily TUI affordances. nav has a spinner, status bar, slash popup, scrollback, and double Ctrl-C quit. It still lacks polished interruption, input history, streaming rendering, richer status, and recovery from common mistakes.

Recommendation

Improve nav's TUI, but only where the work turns it from a demo loop into something Season can actually live in every day. Do not copy Codex's whole boot pipeline for symmetry.

  1. Make conversation continuity real: keep every TUI prompt grounded in the current session transcript, not just the first prompt after `--resume`.
  2. Add session UX: a simple resume picker, `--last`, clearer session identity, and cwd mismatch handling.
  3. Improve operational startup errors: auth, DB open/migration, and state-lock failures should explain the fix before the user sees a raw Rust error.
  4. Then polish daily TUI usability: interrupt a running turn, input history, visible streaming, and status that says what the agent is doing instead of only whether it is ready or working.
