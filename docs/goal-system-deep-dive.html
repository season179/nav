How /goal should
work in nav.

A goal system lets the agent keep working on a big task across multiple turns, automatically, until it's genuinely done — not just until a single turn ends. This page is the full design: what it is, how it fits into nav's architecture, and what needs to be built.

01Concept

The idea in one paragraph.

Normal agent turns stop after each model response. If a task is big — "fix all 47 failing tests" — the agent might fix a few, then stop, and you have to keep saying "keep going". A goal system lets you declare "here is what done looks like", and the agent loops automatically until the objective is genuinely finished, proven against the actual state of the code.

Why this matters Without a goal, every turn is fire-and-forget. With a goal, the agent has a persistent contract: it knows what it committed to, it can measure progress, and it cannot declare victory without evidence. The difference between "I ran some fixes" and "all 47 tests pass" is the difference between a chatbot and an agent.

The goal system is not just a loop. It requires persistence (survives restarts), accounting (tracks tokens and time), steering (injects hidden prompts to guide the model), lifecycle management (states and transitions), and a completion audit (the model must prove it's done, not just say so).

02User experience

The slash commands the user sees.

Six slash commands, all under /goal. They enter through the existing slash_popup.rs dispatcher and route to the goal subsystem.

/goal fix all failing tests
Set a new goal. If a goal already exists and is not complete, confirm before replacing.
/goal
Show current goal: objective, status, tokens consumed, time elapsed, budget remaining.
/goal edit
Open an editor to change the objective mid-flight. Injects a steering prompt so the model adjusts.
/goal pause
Stop auto-continuation. Normal chat still works. The goal waits.
/goal resume
Pick up where it left off. Also unblocks a blocked goal if the user has resolved the blocker.
/goal clear
Remove the goal entirely. All accounting resets.
TUI integration The status bar in app/status_bar.rs should show a compact goal indicator — a gold dot + truncated objective when a goal is active, or nothing when there's none. The composer's slash popup should auto-complete /goal and its sub-commands.
03States

Six states, controlled ownership.

Every goal is always in exactly one state. The key design constraint: the model can only set Complete or Blocked. All other transitions are user-controlled or system-controlled. This prevents the model from accidentally pausing or entering infinite pause/resume loops.

Paused

Blocked
The agent is stuck and needs human input. Set by the model after 3+ consecutive blocked turns, not just because work is hard.
Budget Limited
Token budget exhausted. The agent wraps up, summarizes progress, and reports what remains.
Usage Limited
Rate limit hit. The agent cannot continue until the limit resets or the user upgrades.
Complete
Goal is finished. Every requirement verified against the current state of the code.
04Architecture

Where it lives in nav.

The goal system is a horizontally-cross-cutting feature that touches every layer of the agent. It fits into nav's six harness parts — here's where each responsibility lands.

Tool registry
Three new model-visible tools: get_goal, create_goal, update_goal. Registered in tool_registry/dispatch.rs alongside the existing six tools.
Model
Steering prompts wrapped in <goal_context> XML injected as hidden ResponseInputItems via the existing request builder in model/responses/request.rs.
Context management
Goal context is part of the turn's injected items. Continuation prompts, budget-limit notices, and objective-updated notices are all delivered through the context system.
Guardrails
Objective validation: max 4,000 characters, non-empty, session must be persisted (not ephemeral). Budget enforcement in SQL, not application code.
Agent loop
The continuation check lives in the idle path after TurnComplete. maybe_continue_goal_if_idle() fires when no turn is running and an active goal exists.
Verify
The completion audit is enforced through prompting: the model must cite evidence for every requirement before calling update_goal(status: "complete").
Why this matters Keeping the goal system inside the six existing harness parts — instead of adding a seventh — means the feature composes with everything else. Context compaction still works. The approval policy still applies. Session resume still replays the transcript. No special cases.
05Data model

The ThreadGoal struct.

One goal per session. Stored in the existing SQLite database at nav.db, alongside the session, event, and turn tables. A new thread_goals table, no new database.

CREATE TABLE thread_goals (
  thread_id         TEXT PRIMARY KEY,    -- FK to session row
  goal_id           TEXT NOT NULL,        -- UUID, changes on replace
  objective         TEXT NOT NULL,        -- max 4,000 chars
  status            TEXT NOT NULL DEFAULT 'active',
  token_budget      INTEGER,              -- NULL = unlimited
  tokens_used       INTEGER NOT NULL DEFAULT 0,
  time_used_seconds INTEGER NOT NULL DEFAULT 0,
  created_at_ms     INTEGER NOT NULL,
  updated_at_ms     INTEGER NOT NULL
);

Rust model

struct ThreadGoal {
    thread_id:         Uuid,
    goal_id:           Uuid,        // changes on replace — used for optimistic concurrency
    objective:         String,     // max 4,000 chars
    status:            GoalStatus,  // enum, see below
    token_budget:      Option<i64>, // None = unlimited
    tokens_used:       i64,
    time_used_seconds: i64,
    created_at:        u64,        // millis since epoch
    updated_at:        u64,
}

enum GoalStatus {
    Active,
    Paused,
    Blocked,
    BudgetLimited,
    UsageLimited,
    Complete,
}
Key design constraint goal_id changes every time the goal is replaced. The continuation loop checks goal_id before starting a new turn — if it changed, the goal was replaced mid-flight and the stale continuation is discarded. This is optimistic concurrency without a lock.

Budget enforcement in SQL

The budget check lives in the UPDATE statement itself, not in application code. This eliminates the TOCTOU race between reading tokens_used and writing the update:

-- When setting status = 'active', re-check budget:
status = CASE
  WHEN status = 'budget_limited'
       AND new_status IN ('paused', 'blocked')
    THEN status          -- can't leave budget_limited via pause/block
  WHEN new_status = 'active'
       AND token_budget IS NOT NULL
       AND tokens_used >= token_budget
    THEN 'budget_limited'  -- stay budget_limited if over
  ELSE new_status
END

The database is the source of truth for budget enforcement. Application code never makes budget decisions on its own.

06Continuation

The auto-continuation loop.

This is the core value proposition. When a goal is active and the system is idle, a new turn starts automatically. The check fires after TurnComplete, after ToolCallOutput, and after any event that might leave the system idle.

Read this for crates/nav-core/src/agent_loop/runner.rs — the existing run_agent loop, extended with a maybe_continue_goal_if_idle() call after turn finalization.

Guard conditions

Every condition must be true before a continuation turn starts:

  • Goal system is enabled (compile-time or config flag)
  • No turn is currently running
  • No queued response items for the next turn
  • Session is persisted (not ephemeral / --no-save)
  • Goal exists and status == Active
  • Goal goal_id matches the one we read at check time (optimistic concurrency)

Continuation turn lifecycle

// Pseudocode — lives after turn finalization in runner.rs
async fn maybe_continue_goal_if_idle(&self) -> Result<()> {
    let candidate = self.db.get_thread_goal(self.session_id).await?;
    match candidate {
        Some(goal) if goal.status == GoalStatus::Active => {
            // Build continuation prompt from template
            let prompt = continuation_prompt(&goal);
            let items = vec![goal_context_item(prompt)];

            // Re-read goal from DB to confirm it hasn't changed
            let fresh = self.db.get_thread_goal(self.session_id).await?;
            if fresh.goal_id != goal.goal_id { return Ok(()); }

            // Start a fresh turn with the continuation prompt
            self.start_turn(items).await?;
        }
        _ => {}
    }
    Ok(())
}
Why double-read the goal? Between the first check and the turn start, the user might have run /goal clear or /goal edit. The second read catches this race. The goal_id check is the optimistic concurrency guard — it's cheap and atomic.
07Tools

The three model-facing tools.

Three tools registered in the model's tool list alongside the existing six (read_file, list_files, bash, edit_file, apply_patch, code_search). The model decides when to call them.

01 · READ

get_goal

No parameters. Returns the current goal with status, objective, usage, and budget. The model calls this at the start of a goal turn.

02 · CREATE

create_goal

Parameters: objective (required), token_budget (optional). Fails if a goal already exists. Only called when the user explicitly requests it.

03 · UPDATE

update_goal

Parameter: status — must be "complete" or "blocked". Returns updated goal + optional budget report. Called when the model has verified completion or is genuinely stuck.

The restriction matters

The model cannot set Paused, BudgetLimited, or UsageLimited. Those transitions belong to the user and system. This is deliberate: the model has no context about the user's intent to pause or the system's budget exhaustion. Without this restriction, the model could enter infinite loops of pausing and resuming, or mark itself budget-limited just because it's "taking a break."

Tool response shape The response includes remainingTokens so the model can pace itself. When the status is complete and there's a budget or non-zero time, a completionBudgetReport field instructs the model to report final usage to the user in its next message.
08Steering

Hidden prompts that guide the model.

Steering is how the runtime injects hidden context into the model's input without the user seeing it. Three prompt templates, delivered as hidden <goal_context> blocks at key moments.

continuation.md
Injected at the start of each auto-continuation turn. Contains the objective, budget status, work-from-evidence rules, fidelity rules ("do not substitute a narrower, safer solution"), and the full completion audit protocol.
budget_limit.md
Injected when the token budget is exceeded. Tells the model to wrap up, summarize progress, and not start new work. Deduplicated per goal_id so it's only injected once.
objective_updated.md
Injected when the user edits the goal via /goal edit. Tells the model to adjust to the new objective and abandon work that only served the old one.

The continuation prompt structure

The continuation prompt is the most important template. It has seven sections:

Objective

Repeats the user's goal in XML tags. The model sees its contract every turn.

Continuation behaviour

"This persists across turns; do not shrink the objective." Prevents the model from redefining success around a smaller subset.

Budget status

Tokens used, budget, remaining. Lets the model pace itself.

Work from evidence

"Inspect current state; do not rely on past context." Forces the model to re-verify rather than assume.

Fidelity rules

"Do not substitute a narrower, safer solution." The model cannot redefine the goal to be easier.

Completion audit

Detailed requirement-by-requirement verification protocol. Every sub-goal needs authoritative evidence.

Anti-patterns the prompt forbids
  • Marking complete because the budget is nearly exhausted
  • Marking complete because "stopping work for now"
  • Using "blocked" just because the work is hard or slow
  • Relying on memory instead of inspecting current state
  • Redefining success around a smaller, easier subset
09Accounting

Token and time accounting.

Two concurrent tracking mechanisms, serialized through a Semaphore(1) so that parallel tool completions don't double-count.

Token delta formula

Not all tokens count equally. Cached input tokens are excluded because they're cheap:

token_delta = (input_tokens - cached_input_tokens) + output_tokens.max(0)

Two trackers

Turn accounting

Token delta per turn. Flushed on tool completion, turn end, or external mutation. This is what detects budget exhaustion mid-turn instead of waiting for the turn to finish.

Wall-clock accounting

Elapsed seconds while the goal is active. Flushed on the same triggers, plus before any external mutation. Gives the user a "how long has this been running" readout.

Accounting modes

Different events need different accounting scopes:

ActiveOnly
Only update goals with status = active. Normal tool completions.
ActiveOrComplete
Also update complete goals. Used when the model marks the goal complete — the final turn's tokens still need recording.
ActiveOrStopped
Also update paused, blocked, usage_limited. Used when an external stop happens mid-turn — in-flight tokens still count.
Why tool-level granularity? Turn-level accounting alone would miss budget exhaustion mid-turn. Tool-level accounting lets the system inject budget steering immediately after the budget is crossed, rather than waiting for the turn to end after potentially significant additional token usage.
10Events

Runtime events and lifecycle hooks.

The session emits goal lifecycle events at specific points. Each event maps to a handler that updates accounting, injects steering, or starts continuation. These extend the existing AgentEvent enum.

TurnStarted
Capture active goal ID and token baseline. Skip if no goal exists.
ToolCompleted
Account token delta. May inject budget steering if threshold crossed. Emit terminal metrics.
TurnFinished
Final accounting for the turn. Clear continuation turn ID. Check maybe_continue_goal_if_idle.
GoalToolCompleted
update_goal finished. Account tokens but suppress budget steering — the model already knows.
UsageLimitReached
Set goal to UsageLimited. Notify user in TUI.
ExternalGoalSet
User ran /goal slash command. Update accounting state. Inject objective-updated steering if goal changed. Maybe continue.
ExternalGoalClear
User ran /goal clear. Clear all runtime state for this goal.
SessionResumed
Session resumed from save. Restore accounting for active goals. Emit resumed metric.
Composition with existing events These events don't replace the existing TurnComplete, ToolCallOutput, etc. They layer on top. The TUI sees the same events it always did; goal-aware handlers are additional subscribers that react when a goal is present and ignore events when there's none.
11Build order

What to build first.

If you're implementing this feature, here's the minimum viable set of components in the order that makes the dependencies work.

Phase 1
3files

Data model, persistence, and three tool definitions. At this point the model can set and read goals, but there's no auto-continuation.

Phase 2
2files

Continuation loop + continuation prompt template. Now goals auto-continue. This is where the feature becomes useful.

Phase 3
4files

Token accounting, budget enforcement in SQL, budget-limit steering, and the remaining slash commands. The feature is now production-ready.

Phase 1: Foundation

1. Data model
Define ThreadGoal struct with GoalStatus enum, objective, budget, usage counters, timestamps. One goal per session.
2. Persistence
Add thread_goals table to the existing migration in context/session/mod.rs. Implement atomic upsert/insert/update with optimistic concurrency via goal_id.
3. Tool definitions
Register get_goal, create_goal, update_goal in tool_registry/dispatch.rs. Restrict update_goal to complete/blocked only.

Phase 2: The loop

4. Continuation loop
Add maybe_continue_goal_if_idle() after TurnComplete in runner.rs. Check guards, build continuation prompt, start new turn.
5. Continuation prompt
Write continuation.md template with objective, budget status, audit rules. Store in crates/nav-core/src/context/templates/ or inline as a const.

Phase 3: Production

6. Token accounting
Track token delta and wall-clock time per turn in runner.rs. Flush to thread_goals on tool completion and turn end.
7. Budget enforcement
Add the CASE check to the SQL update statement. Inject budget_limit.md steering when budget is crossed. Deduplicate by goal_id.
8. Slash commands
Wire /goal, /goal pause, /goal resume, /goal edit, /goal clear through slash_popup.rs and bottom_pane/. Show goal status in the status bar.
9. Session resume
Goals must survive session restart. On --resume, restore accounting state from the DB and emit a goal snapshot so the user sees the goal is still there.
The hard parts Three things to get right: (1) accounting concurrency — tool completions happen in parallel and you must not double-count; (2) the continuation loop — handle the case where the goal changes between the check and the turn start; (3) the completion audit — without strong prompting, the model will mark goals complete prematurely.
