A reader's tour · nav

How nav works, in plain English

nav is a small Rust coding agent. It reads files, edits them, runs commands, and talks back to you in a terminal chat. This page walks through the code one piece at a time, in the order that makes it easiest to learn. No prior agent experience needed.

1The big picture in 60 seconds

A coding agent is a loop. The loop is simple enough to describe in one paragraph: the user types a prompt; nav sends it to a model; the model either replies with text or asks nav to run a tool; nav runs the tool, hands the output back to the model, and asks again; repeat until the model is done. That's it. The rest of the code exists to make that loop pleasant to watch, easier to audit, and possible to resume tomorrow.

Everything in nav-core is in service of that loop. Everything in nav-tui is about painting it on screen. If a piece of code doesn't fit one of those two roles, treat that as a smell.

nav splits into two pieces you'll spend your time in:

nav-core (the library)

The loop, the tools, the model transport, the session database. No screen, no keyboard. Just produces a stream of AgentEvents that anyone can consume.

nav-tui (the terminal UI)

Takes the event stream and renders the chat: messages above, composer below, status bar at the bottom. Built on ratatui + crossterm.

A tiny binary nav-cli wires those two together. When stdout is a terminal and --json-events is not set, it launches the TUI. Otherwise — for example when you pass --json-events or pipe stdout to a file — it emits raw events as JSON, one per line.

3A suggested reading order

Open the files in this order. Each one will make more sense than if you'd opened them alphabetically.

  1. crates/nav-core/src/cli.rs — the CLI options. Tiny. Shows what knobs exist.
  2. crates/nav-core/src/agent.rs — start at run_agent. The whole loop is one function.
  3. crates/nav-core/src/tools/mod.rs — the dispatcher; five tools, one match arm each.
  4. crates/nav-core/src/tools/fs.rs — the workspace sandbox. Notice resolve_inside.
  5. crates/nav-core/src/responses/mod.rs — turns the loop's request into a Responses API body.
  6. crates/nav-core/src/session/mod.rs — SQLite event log behind --resume.
  7. crates/nav-tui/src/lib.rs — the TUI's run(): tokio::select! over keys + agent events.
  8. crates/nav-tui/src/widget.rs + cells.rs — how each event becomes a row on screen.
  9. crates/nav-tui/src/streaming.rs — the helper meant to make live streamed text not flicker.
  10. crates/nav-tui/src/bottom_pane/ — composer and slash-command overlay.

4The agent loop, step by step

crates/nav-core/src/agent.rs — function run_agent.

Open agent.rs and find run_agent. Strip out the bookkeeping and you're left with this skeleton: 

for _ in 0..args.max_turns {
    // 1. ask the model
    let body = responses::response_body(args, cwd, &input);
    let mut stream = transport.create(body).await?;

    // 2. read its streamed reply; emit deltas as we see them
    while let Some(event) = stream.next().await {
        emit_stream_events(&event, &events, session);
        if collector.push_event(&event, "Responses API")? { break; }
    }
    let envelope = collector.finish("Responses API")?;

    // 3. did the model ask to run any tools?
    let calls = responses::function_calls_from(&envelope)?;
    if calls.is_empty() {
        finalize_turn(...);   // model is done — we're out
        return Ok(());
    }

    // 4. run each tool, append the model's call + our output to `input`
    for call in calls {
        emit(ToolCallStarted { call_id, name, arguments });
        let (text, is_error) = tools::run_tool(...).await;
        input.push(function_call_output);
        emit(ToolCallOutput { call_id, output: text, is_error });
    }

    finalize_turn(...);  // loop and ask again
}

One "turn" = one model call + zero-or-more tool calls. The function exits as soon as the model returns a turn without any tool calls. max_turns exists as a guardrail so a runaway model can't burn forever.

The model does not call local code directly. It asks for a tool in JSON; tools::run_tool dispatches by name; and Rust decides what actually runs. The file/search/edit tools enforce workspace path checks. bash is deliberately broader: it runs sh -c in the workspace with a timeout and captured output, but it is not a filesystem sandbox.

5The AgentEvent bus

crates/nav-core/src/agent.rs — enum AgentEvent.

Instead of println!-ing as it goes, the loop emits one AgentEvent per observable step. Every frontend — TUI, NDJSON pipe, future Slack bot — reads the same six variants:

AssistantMessageDelta
A chunk of streaming text the model is still typing. Transient — paint it, but don't save it.
AssistantMessageDone
The full final text for one assistant message. The durable record.
ToolCallStarted
"I'm about to run bash with these arguments." Fires before the tool actually runs.
ToolCallOutput
The tool's result text. For bash, that includes status, stdout, and stderr. is_error lets the UI colour failures red.
TurnComplete
End of one model turn, with token usage. The TUI clears the spinner here.
Error
Something failed, or the UI needs to show a problem. Errors produced by the agent loop end that run.

Two flavours: durable events (everything except AssistantMessageDelta) are written to the SQLite log; the delta is paint-only and never persisted. That distinction is encoded in one method, AgentEvent::is_durable.

Splitting "live render chunk" from "official record" means --resume can replay the conversation from durable events alone, even though you saw it stream in real time the first time around. No special-case code, no second log.

6The five tools the model can call

crates/nav-core/src/tools/ — mod.rs, fs.rs, shell.rs.

The model is told, in JSON-Schema form, that it has exactly five functions to call. They are deliberately primitive — together they cover "inspect, find, change, verify."

read_file

Read one file. Relative paths only. Directories rejected.

list_files

List a directory. Returns names sorted alphabetically; directories get a trailing /.

bash

Run a shell command via sh -c, with a timeout, capturing stdout+stderr+exit.

edit_file

If old_str is empty, create the file. Otherwise, replace exactly one occurrence of old_str with new_str. Two matches = error.

code_search

Shell out to rg (ripgrep). Exit code 1 (no matches) is treated as success — that's not an error.

(That's all.)

Notice what's not here: no delete_file, no git_commit, no http_get. Less surface = easier to reason about.

The workspace sandbox

Every path used by read_file, list_files, edit_file, and code_search goes through resolve_inside (or resolve_create_path, for new files). These reject:

  • absolute paths (/etc/passwd)
  • parent traversal (../escape.txt)
  • symlinks that escape the workspace
  • (on create) any symlink anywhere in the new path
The model is good at writing plausible-looking code, but it's also good at writing plausible-looking ../../etc/passwd. Path resolution doesn't trust the model's claim that a path is "relative"; it canonicalize()s the resolved path and checks it still starts with the workspace root. Treat bash separately: it is a powerful escape hatch, not part of this path sandbox.

7Talking to the model

crates/nav-core/src/responses/ — mod.rs, websocket.rs, sse.rs, types.rs.

The model lives behind OpenAI's Responses API. nav speaks to it two ways:

  • WebSocket (default) — one socket per turn, lower latency, cleaner framing.
  • SSE (--transport sse) — plain HTTP with text/event-stream. Slower, but easier to debug because you can curl it.

Both implementations boil down to the same shape: "send the request body, then push every event you receive into an unbounded channel." The agent loop reads from that channel as a Stream and doesn't care which transport is producing it.

The interesting glue is the ResponseCollector in responses/mod.rs. As stream events come in, it accumulates output items and waits for the terminal response.completed event. Then finish merges streamed items with whatever the final envelope claimed, so the loop sees one consistent ResponseEnvelope regardless of how the bytes arrived.

Why two transports? Because nav is partly an educational project. WebSocket is what you want in production; SSE is what you want when you're learning what a Responses event stream actually looks like on the wire.

Authentication, briefly

auth.rs resolves to one of two backends:

  • ChatGPT (default) — reads ~/.codex/auth.json (written by codex login) and uses the OAuth access token against chatgpt.com/backend-api/codex.
  • API key (--auth api-key) — uses OPENAI_API_KEY against the public OpenAI endpoint.

After resolution, the transport code only sees the resolved endpoints and a bearer token.

8Sessions and resume

crates/nav-core/src/session/mod.rs.

Every run is persisted to a SQLite file (default ~/Library/Application Support/nav/nav.db on macOS). Three main tables, plus a small schema_version table for migrations:

  • session — one row per run, with rollups (cwd, model, token totals).
  • event — one row per durable AgentEvent, JSON-encoded, with a per-session sequence number.
  • turn — one row per model turn, with usage + (optional) cost.

--list-sessions reads session with a rollup query. --resume <id> loads every event for that session in order and hands them to rebuild_responses_input, which translates each one back into the wire format the Responses API expects for "here is the conversation so far."

Notice nav never computes cost from tokens × pricing. The response object this code parses gives token usage, not a per-response dollar amount, so complete_turn is called with None, each turn records cost_source = 'unreported', and the listing shows —. The schema is ready for a provider or separate lookup that reports real cost later, but nav refuses to fake the number.

9The TUI: how the chat appears on screen

crates/nav-tui/src/lib.rs — function run.

The TUI lays out three regions, top to bottom:

nav · gpt-5.5 · ~/code/nav · session 01JFK7…
Type a prompt to begin. Slash commands:
/quit exit
/clear start a new transcript
/sessions shown in popup, not wired yet
› what does this repo do?
• Let me look around.
• list_files {"path":"."}
└ Cargo.toml, README.md, crates/, …
• It's a small Rust coding agent…
› Ask nav to do anything
gpt-5.5 · ~/code/nav · main · Ready

The three regions, in code terms:

  • Scrollback — ChatWidget in widget.rs. A Vec<Box<dyn HistoryCell>> painted top to bottom.
  • Composer — BottomPane in bottom_pane/mod.rs. Owns a Composer and an optional overlay (currently just SlashCommandPopup).
  • Status bar — StatusBar in status_bar.rs. One row: model · cwd · branch · Ready or Working 4s.

The TUI's main loop

run in nav-tui/src/lib.rs is a single tokio::select! over three sources: 

tokio::select! {
    Some(ev) = agent_rx.recv() => chat.ingest(ev);   // model said something
    Some(app) = app_rx.recv()  => …;                 // /quit, /clear, Submit
    _ = sleep(80ms) => {                             // periodic redraw + key poll
        spinner_tick += 1;
        while event::poll(0)? { handle key … }
    }
}

Every iteration draws the frame, then waits for the next thing to happen. The 80ms tick advances the spinner and polls the keyboard; the two channels deliver the things that should change what's on screen. When the user hits Enter, we spawn the agent loop on a background task and let its events flow back through agent_rx.

Why the background task? Because run_agent is async and spends most of its time waiting for the model. If the TUI awaited it directly, the spinner would freeze and keystrokes would back up. Spawning it means the draw loop stays responsive — the agent's progress arrives as events on a channel, same as any other input.

Translating events into cells

ChatWidget::ingest in widget.rs is a small match that maps each AgentEvent variant to a concrete cell type in cells.rs:

AssistantMessageDone
→ AssistantMessageCell, gutter glyph •, green.
ToolCallStarted
→ ToolCallCell, gutter glyph •, yellow. Shows name {json args}.
ToolCallOutput
→ ToolOutputCell, gutter glyph └, dim grey (red if is_error).
Error
→ ErrorCell, red.
User input (not an event)
→ UserMessageCell, gutter glyph ›, bold cyan.
AssistantMessageDelta / TurnComplete
Ignored by the widget today. The final assistant text appears on AssistantMessageDone; TurnComplete flips the status bar.

10Streaming text without flicker

crates/nav-tui/src/streaming.rs.

Models stream text. Important current-state note: ChatWidget currently ignores AssistantMessageDelta, so the TUI shows the final assistant message when AssistantMessageDone arrives. StreamController is the helper that keeps live text stable once those deltas are wired into a live assistant cell.

If a frontend feeds deltas into AssistantMessageCell::streaming(), simply appending every delta and re-rendering the whole thing each frame would be readable but jumpy: a half-written table row reflows as cells widen, code fences appear before their closing backticks, and so on.

StreamController solves that with one rule:

Lines end up in one of two buckets. A line that ends with a real newline and isn't inside an open markdown construct (fenced code block, table) moves to stable. Everything else stays in tail. Stable lines are rendered as-is and never reflow. The tail re-renders every frame and absorbs the wobble.

The partition is computed by scanning lines once and tracking a tiny state machine: Outside → Fence(`` ``` ``) / Fence(~~~) / Table → back to Outside. When the model finishes, finalize() flips a flag and the whole buffer becomes stable in one step.

This isn't a markdown renderer. It's just enough markdown awareness to know "the line I'm looking at isn't safe to commit yet because we're mid-table." The actual rendering is plain monospace — the partition just keeps live text from jumping around.

11The composer and slash popup

crates/nav-tui/src/bottom_pane/composer.rs and slash_popup.rs.

The Composer is a small multi-line text editor with bash-style key bindings:

  • Enter submits; Shift+Enter inserts a newline.
  • Ctrl+U clears to the start of line. Ctrl+W deletes the previous word.
  • ↑ / ↓ at the top/bottom row recall earlier prompts from this session.
  • Esc dismisses overlays (and the slash popup when it's open).

Overlays: who sees the key first

The BottomPane can host an overlay on top of the composer. Today there's exactly one (SlashCommandPopup), but the routing is general: 

fn handle_key(&mut self, key) {
    if let Some(view) = self.view.as_mut() {
        match view.handle_key(key, &mut self.composer) {
            Handled   => { /* stop here */ }
            Unhandled => { /* fall through */ }
        }
    }
    self.composer.handle_key(key)
}

The popup auto-opens when the buffer starts with / and has one line, and auto-closes when that's no longer true. Today /quit and /clear are the commands wired up in run(). The popup also lists /help, /resume, and /sessions as placeholders; selecting them puts the text in the composer, but pressing Enter sends them as a normal prompt until run() handles them. slash_popup_suppressed remembers "the user just hit Esc to dismiss me; don't pop back open on the next keystroke."

12Try it yourself

The fastest way to confirm any of the above is to run it and watch. Two prompts that exercise the main flow and the raw event stream: 

cd ~/code/your-project
nav "list the files and read README.md"

A typical run shows list_files arrive as a ToolCallStarted cell (yellow), its output as a ToolOutputCell (grey, attached with └), then read_file, then a final assistant message (green) summarising what it found. Compare that against the six events in section 5, but remember that not every run emits every variant, and the current TUI does not display AssistantMessageDelta chunks. 

nav --json-events "explain this repo" > events.ndjson

Same kind of agent run, but the TUI is off. Each line of events.ndjson is one AgentEvent as JSON. This is the wire format the CLI exposes for non-Rust frontends — and it's also a fantastic debugger when something looks wrong in the TUI: was the event missing, or just rendered wrong?
